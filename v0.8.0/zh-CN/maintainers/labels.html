ZeroClaw Docs

标签

每个在 PR 和 Issue 中使用的标签的单一参考来源。权威来源：

  • .github/labeler.yml：由 actions/labeler 使用的路径标签配置
  • .github/label-policy.json：贡献者等级阈值
  • 本页：定义、行为，以及哪些是自动化的、哪些是手动的

当定义发生冲突时，请先更新源文件，然后同步此页面。

所有权边界

标签是可移植的元数据。它们应该说明这是什么类型的工作、涉及哪部分代码区域、审查的风险有多大，以及是否需要对过期策略或分类策略进行特殊处理。

在添加项目看板（Project board）自动化时，应将其作为自动化的规划看板使用，而非第二个 PR 审查队列。该看板应回答节奏较慢的规划类问题：有哪些任务可以着手处理、由谁负责、属于哪个跟踪器或里程碑，以及哪些任务受阻。而原生的 GitHub PR 状态则应继续回答节奏较快的审查与合并类问题。

根据更新频率保持拆分：

  • 标签拥有持久的分类信息：工作类型、范围/组件、审查风险、已度量的 PR 规模以及过期豁免。
  • 当这些字段得到积极维护时，项目看板字段适用于议题规划阶段、当前负责人或管理者路径、依赖状态、过期豁免原因以及路线图分组。
  • GitHub PR 原生状态负责管理快速变化的审查状态：审查决定、必需的检查、可合并性、冲突以及过期的批准。

看板应当减少维护者的工作量。如果某个字段在每次 PR 推送或审查后都需要手动维护，建议改用标签、里程碑或 GitHub 原生状态。

标签可以提示可能的归属，但它们并不等同于归属。channel:*、provider:*、tool:*、security 或 docs 标签标识了可能需要关注的部分。贡献者可见的所有者来源规则参见 Project board contract。

使用指派人来标识正在进行的工作。当尚无人开始实现时，使用领域负责制来分配路由职责。项目看板约定定义了已认可的负责人来源和路由结果。

规范拼写

对于范围化的模块标签，请使用无空格的实时模块拼写形式：provider:openai、channel:telegram、security:policy 等类似标签。size、risk 和大多数 type 系列目前在冒号后保留一个空格：size: XS、risk: low、risk: medium、risk: high 和 type: docs。

诸如 provider: openai、channel: telegram 或 tool: shell 这类遗留的重复标签都是待清理对象。未来针对 size、risk 或 type 标签采用无空格写法同样属于迁移问题，而非随意重命名。删除前应将未关闭的 issue/PR 迁移到规范标签上。请勿删除仍有未关闭引用的标签、大范围重命名标签族，或在未经维护者就该清理批次做出决定的情况下移除 stale-policy 标签。

自动化契约

实时 PR 标签自动化按来源拆分。pr-path-labeler.yml 会在 PR 打开、重新打开以及每次推送更新时，依据 .github/labeler.yml 运行 actions/labeler。由于该工作流使用 sync-labels: true，归属于 .github/labeler.yml 的标签会根据当前 PR 文件集重新计算：匹配的路径标签会被添加，而不再匹配的路径标签会被移除。

Dependabot 还会根据 .github/dependabot.yml 为自己创建的 PR 添加已配置的标签：Cargo 更新会获得 dependencies；GitHub Actions 和 Docker 更新会获得 ci 和 dependencies。这些标签属于 Dependabot PR 的初始元数据，而不是同步的 path-labeler 约定。

目前 .github/labeler.yml 仅管理路径和范围标签，例如 docs、ci、channel、provider:openai 和 tool:file。它不管理 risk:*、size:*、type:*、贡献者等级、状态、解决方案、过期或待领取标签。

如果稍后添加风险或规模自动化，则应在每次推送的 PR 更新时重新计算，以便标签持续描述正在审查的实际差异。风险自动化必须将 risk: manual 视为一种覆盖项，在维护者移除该覆盖项之前，阻止对该 PR 进行后续的自动化风险替换。

清理协议

标签清理是维护者的操作，而非常规 PR 审查的附带结果。

使用以下序列：

  1. 操作前请刷新实时标签使用情况。
  2. 将候选项划分为无历史删除、无打开项的重复删除、优先迁移的活跃标签，以及策略保留项。
  3. 对于带有未关闭引用的标签，为每个未关闭的 issue/PR 添加规范标签，移除旧标签，确认旧标签的未关闭引用数为零，然后将其删除。
  4. 在进行模块标签清理时，请勿删除治理标签、过期策略标签、贡献者等级标签或 GitHub 默认标签。

每个实时清理批次都需要维护者对正在更改的标签和 issue/PR 引用进行明确批准。

类型标签

类型标签用于标识高层次的工作类别。它们与路径标签（如 docs、ci 或 dependencies）是相互独立的。

type:rfcRFC 议题或提案；在活动期间免于陈旧关闭

风险标签

对于 PR，风险标签描述了正在审查的实际差异：涉及的路径、行为变更、安全边界暴露以及回滚难度。对于 issue，风险标签描述了基于报告的潜在修复影响范围，有助于分流审查者的深度和贡献者的匹配度，并且可能在具体 PR 显示实际实现路径后发生变化。当前为手动应用。未来的风险自动化应遵循自动化契约。

标签含义
risk: low未触及高风险路径，属于小改动
risk: medium对 crates/*/src/** 进行行为变更，但不影响边界或安全性
risk: high触及高风险路径，或进行大型安全相关变更
risk: manual维护者覆盖，用于冻结自动风险重新计算

高风险路径（规范集；其他维护者页面引用此列表）：crates/zeroclaw-runtime/src/**、crates/zeroclaw-gateway/src/**、crates/zeroclaw-tools/src/**、crates/zeroclaw-runtime/src/security/**、.github/workflows/**。

当不确定时，视为高风险。

状态标签

跟踪 RFC 和跟踪工作项的生命周期状态。除非有维护中的工作流另作说明，否则手动应用。

标签描述
status:accepted由团队批准的 RFC 或工作项。但这本身并不能使该问题免于过期处理。
status:blocked工作有效，但正在等待外部依赖、维护者决策或关联的前置条件。在阻塞因素被记录且未解决期间，可豁免标记为过时。请勿针对同一阻塞因素与 status:no-stale 搭配使用。
status:in-progress一个开放的 PR 正在积极处理此 issue。在过期检查时请与实时的 PR 状态进行核对；该标签在 PR 关闭后并非永久豁免。
status:stale作者在过期窗口内无活动；如不刷新可能会被关闭
status:no-stale针对已接受或其他长期存在、且尚未受其他过期排除规则保护的工作的显式过期豁免。目标策略：仅当项目看板约定具备贡献者可见的过期豁免原因和负责人路径时使用。活跃的发布跟踪器及活跃的 RFC 或设计跟踪器在保持活跃期间，可将跟踪器本身作为可见原因和管理界面；当里程碑关闭、跟踪器偏离实时状态、RFC 达成决策、被取代或关闭，或该 issue 不再代表活跃的项目决策界面时，应重新审视它们。对于缺少这些事实的现有豁免，应在过期清理停止认可它们之前进行审计和修复。

分辨率标签

解决方案标签用于说明某个 issue 或 PR 被关闭或移出活动队列的原因。它们代表最终结果，而非生命周期状态标签，并且应当包含足够的注释上下文，以便后续维护者理解该决策。

标签目的
wontfix有效的请求，或者明确说明项目选择不予采纳的报告。请简要说明理由，不要无声关闭。
invalid无法作为缺陷、功能请求、支持事项、RFC 或跟踪的项目工作处理。请说明不匹配之处或缺失的需求。
duplicate与另一个已跟踪的问题或 PR 属于同一底层问题。在关闭或重定向讨论之前，请链接到规范的目标。

在维护者批准的标签迁移包明确定义具体的重命名、别名或删除计划之前，请勿创建或应用诸如 status:wont-do 或 status:wont-fix 之类的拟议终端标签。当前看板级“Won’t Do“概念使用的有效标签为 wontfix。

取代（Superseding）是一种替换流程，目前并不是一个生效的标签。在后续批准的迁移数据包创建或映射取代标签之前，请使用 Superseding PRs 中的替换规则和归属要求。

标签目的

维护触发器

在以下情况时更新此页面：

  • 向源代码树中添加了新的通道、提供程序或工具（路径标签需要新增条目）。
  • 标签策略或阈值已更改。
  • 新的分诊工作流出现，或旧的被移除。

自动化状态备注（“目前手动应用”）被特意包含在内，以便未来的维护者不会误以为工作流的缺失意味着标签层级不存在。
