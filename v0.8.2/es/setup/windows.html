ZeroClaw Docs

Windows

Instalar, actualizar, ejecutar como tarea programada de Windows y desinstalar en Windows 10 / 11.

Si está ejecutando WSL2, puede seguir la configuración de Linux en su lugar; install.sh se ejecuta sin cambios en WSL.

Nota sobre setup.bat. Los fallos de detención total de #6118 (el desbordamiento de espacio en disco de set /a de 32 bits y el error de análisis de if/else por paréntesis sin escapar) se corrigieron en #6137 y se incluyen en la v0.7.4 y posteriores. Las versiones actuales se completan con normalidad. Queda una advertencia: setup.bat --prebuilt aún comprueba cargo antes de llegar a la rama precompilada, por lo que la ruta de precompilado manual (Opción 1 a continuación) es la verdadera instalación sin Rust. Compilar desde el código fuente (Opción 3) también funciona.

Instalar

Opción 1: Binario precompilado (recomendado)

Descarga el archivo zip de la última versión para Windows, extrae zeroclaw.exe y colócalo en tu PATH.

Desde un símbolo del sistema de PowerShell:

# Idempotente: volver a ejecutar este bloque no tiene efecto cuando zeroclaw ya está
# instalado en la última versión y en el PATH del usuario. Después de que una versión
# se actualice, la comprobación de versión falla y el lado de instalación se ejecuta de nuevo.
$ver = (Invoke-RestMethod 'https://api.github.com/repos/zeroclaw-labs/zeroclaw/releases/latest').tag_name.TrimStart('v')
$dst = "$env:USERPROFILE\.zeroclaw\bin"
$exe = "$dst\zeroclaw.exe"

$current = if (Test-Path $exe) {
    ((& $exe --version 2>$null) | Select-String -Pattern '\d+\.\d+\.\d+').Matches.Value
} else { '' }

if ($current -ne $ver) {
    $url = "https://github.com/zeroclaw-labs/zeroclaw/releases/download/v$ver/zeroclaw-x86_64-pc-windows-msvc.zip"
    New-Item -ItemType Directory -Force -Path $dst | Out-Null
    Invoke-WebRequest -Uri $url -OutFile "$env:TEMP\zeroclaw.zip" -UseBasicParsing
    Expand-Archive -Force -Path "$env:TEMP\zeroclaw.zip" -DestinationPath $dst
}

$userPath = [Environment]::GetEnvironmentVariable('Path', 'User')
if (($userPath -split ';') -notcontains $dst) {
    [Environment]::SetEnvironmentVariable('Path', "$dst;$userPath", 'User')
}
if (($env:Path -split ';') -notcontains $dst) {
    $env:Path = "$dst;$env:Path"
}

& $exe quickstart

El archivo zip incluye un binario autónomo, sin necesidad de Rust toolchain ni Visual Studio Build Tools.

Después de la instalación, verifique:

zeroclaw --version    # matches the latest release

Opción 2: setup.bat (desde una versión)

setup.bat --prebuilt

Flags:

BanderaComportamiento
--prebuiltDescargar el binario precompilado desde GitHub Releases (lo más rápido una vez disponible; el script actual todavía comprueba primero cargo)
--minimalCompilar solo el núcleo (sin canales, sin hardware)
--standardConstruye con canales comunes (Telegram, Discord, Slack, Matrix)
--fullCompilar todo

⚠️ Problema conocido (actual). setup.bat --prebuilt aún comprueba cargo antes de llegar a la rama prebuilt, por lo que la Opción 2 no respeta la promesa de no usar Rust. Si no tienes una cadena de herramientas de Rust, usa la Opción 1 anterior.

Histórico (anterior a v0.7.4). Las versiones anteriores tenían dos fallos de detención total y una discordancia en el comando de incorporación reportados en #6118: un desbordamiento de espacio en disco de 32 bits en set /a (Invalid number. Numbers are limited to 32-bits of precision.), un error de análisis de if/else por paréntesis sin escapar (.[0m was unexpected at this time.) y un prompt final de zeroclaw init. Todos se corrigieron en #6137 (v0.7.4 / 0.7.5 / 0.8.0); las versiones actuales imprimen zeroclaw quickstart y se completan con normalidad.

Opción 3: Desde el código fuente

Requiere Rust (rustup) y las herramientas de compilación de Visual Studio:

git clone https://github.com/zeroclaw-labs/zeroclaw
cd zeroclaw
cargo install --locked --path .
zeroclaw quickstart

Opción 4: Scoop (actualmente desactualizado)

⚠️ El manifiesto de Scoop en el repositorio está fijado a la v0.5.9 (23 versiones de parche por detrás de master). Hasta que un hook de CI en el momento del lanzamiento lo actualice, es preferible usar la Opción 1 o 3. Si usas Scoop y encuentras problemas, abre un PR contra dist/scoop/zeroclaw.json.

scoop install zeroclaw     # currently installs an older release; see warning above
zeroclaw quickstart

Opción 5: Docker

ZeroClaw publica una imagen de contenedor de Linux en ghcr.io/zeroclaw-labs/zeroclaw:latest (y :vX.Y.Z para versiones etiquetadas). En Windows, ejecútala mediante Docker Desktop o mediante sudo apt install docker.io dentro de una distribución WSL; ambas opciones funcionan y el comportamiento del contenedor es idéntico.

Inicio rápido:

# Volumen persistente para config + workspace; el directorio de datos de ZeroClaw dentro del contenedor es /zeroclaw-data
docker run -d --name zeroclaw `
  --restart=unless-stopped `
  -p 42617:42617 `
  -v zeroclaw-data:/zeroclaw-data `
  ghcr.io/zeroclaw-labs/zeroclaw:latest

# Observa los registros de la primera ejecución
docker logs -f zeroclaw

# Comprobación de estado (no requiere autenticación)
curl http://localhost:42617/health

# (Opcional) emparejar un cliente. NOTA: la imagen publicada usa por defecto
# `require_pairing = false`, por lo que de forma predeterminada no se emite ningún código de emparejamiento
# y `/api/*` acepta solicitudes sin autenticación. Para habilitar el emparejamiento,
# sobrescribe la configuración (establece `require_pairing = true` en
# /zeroclaw-data/.zeroclaw/config.toml dentro del contenedor) y reinicia;
# entonces se imprime un código de un solo uso en stdout en el primer inicio, tras lo cual
# los clientes lo envían mediante POST a `/pair`:
curl -X POST http://localhost:42617/pair -H 'X-Pairing-Code: <code-from-logs>'

Datos de la imagen (verificados con ghcr.io/zeroclaw-labs/zeroclaw:latest):

  • Base: gcr.io/distroless/cc-debian13:nonroot (etapa de release; la etapa dev es debian:trixie-slim)
  • ENTRYPOINT ["zeroclaw"], CMD ["daemon"]: ejecutar sin argumentos inicia el daemon y el gateway
  • EXPOSE 42617: tanto el daemon como el gateway escuchan en este puerto
  • Directorio de datos: /zeroclaw-data (configuración: /zeroclaw-data/.zeroclaw/config.toml, espacio de trabajo: /zeroclaw-data/workspace). Monta aquí un volumen con nombre o un bind para la persistencia; ten en cuenta que esto no es /root/.zeroclaw.
  • Emparejamiento: la imagen publicada usa por defecto require_pairing = false, por lo que /api/* acepta solicitudes sin autenticación de forma predeterminada. Cuando el emparejamiento está habilitado (establece require_pairing = true en /zeroclaw-data/.zeroclaw/config.toml y reinicia), el daemon imprime un código de un solo uso en stdout en el primer inicio, y los clientes luego lo envían mediante POST a /pair con el encabezado X-Pairing-Code antes de que cualquier endpoint autenticado responda.
  • Panel web: empaquetado y servido por defecto en la imagen publicada. La imagen establece gateway.web_dist_dir = "/usr/share/zeroclawlabs/web/dist" e incluye allí el frontend compilado, por lo que el gateway sirve el fallback de la SPA de forma inmediata. Los assets se encuentran fuera del punto de montaje /zeroclaw-data, de modo que un montaje de volumen -v …:/zeroclaw-data no puede ocultarlos (ref #6400).

Compila desde el código fuente usando el Dockerfile incluido:

git clone https://github.com/zeroclaw-labs/zeroclaw
cd zeroclaw
docker build -t zeroclaw:local -f Dockerfile.debian .

Verificado en Windows + Docker:

  • El comportamiento del contenedor coincide con el de Linux. Descargué y ejecuté ghcr.io/zeroclaw-labs/zeroclaw:latest en WSL Debian en Windows 11 build 26200.8313. La imagen se inicia correctamente, el gateway escucha en :42617 y /health devuelve un JSON válido. Con require_pairing = true establecido en la configuración y el contenedor reiniciado, el flujo de código de emparejamiento en /pair también funciona como se documenta.
  • Docker sin Docker Desktop. wsl --install para habilitar WSL2 y luego sudo apt install docker.io dentro de la distribución de WSL te proporciona el demonio directamente; verificado para descargar y ejecutar la imagen publicada sin modificaciones.

Buenas prácticas del lado del host: orientación general sobre Docker + WSL2, no afirmaciones de runtime específicas de zeroclaw. Procedente de Microsoft Learn y de la documentación propia de Docker cuando corresponde:

  • Montajes de volumen. El montaje de tipo bind de rutas del lado de Windows (-v C:/Users/...:/zeroclaw-data) en un contenedor Linux cruza el límite del sistema de archivos WSL2 ⇄ Windows; Microsoft documenta la disposición y las implicaciones de rutas entre sistemas operativos en la referencia de sistemas de archivos WSL. Es preferible usar volúmenes con nombre de Docker (-v zeroclaw-data:/zeroclaw-data) o almacenar el espacio de trabajo dentro del sistema de archivos WSL (\\wsl$\Debian\home\...) para obtener un rendimiento casi nativo.

  • Redes. La red predeterminada de WSL2 usa NAT, por lo que los servicios del contenedor son accesibles desde Windows mediante localhost:<port> tras el reenvío con -p (verificado en Windows 11 + WSL2). Si necesitas acceder al contenedor desde otra máquina en la LAN, o ejecutar configuraciones multicontenedor donde importe el DNS entre contenedores, cambia al modo reflejado según la referencia Mirrored mode networking de Microsoft, agregando lo siguiente a %USERPROFILE%\.wslconfig:

    [wsl2]
networkingMode=mirrored

  • Daemon en Docker, no en el Programador de tareas. Dentro del contenedor no existe el Programador de tareas de Windows. Usa la política de reinicio de Docker, --restart=unless-stopped como en el ejemplo anterior, para el arranque en modo daemon. La imagen publicada se ejecuta como PID 1 / usuario nonroot; el contenedor es el servicio; no ejecutes zeroclaw service install dentro de él.

  • Sandbox de skills mediante el socket de Docker del host. El sandbox de ejecución de skills de ZeroClaw puede invocar Docker. Si estás ejecutando el propio ZeroClaw en un contenedor y quieres que el sandboxing de skills también use Docker, monta el socket de Docker del host para que los contenedores hijos se ejecuten en el daemon del host en lugar de anidar Docker dentro de Docker:

    # PowerShell / cmd.exe: usa una sola barra inicial.
# Git Bash / MINGW: usa //var/run/docker.sock para evitar la reescritura de rutas de MSYS.
-v /var/run/docker.sock:/var/run/docker.sock

    Tenga en cuenta que montar el socket de Docker otorga acceso al host equivalente a root para cualquier cosa dentro del contenedor; la página Protect the Docker daemon socket de Docker cubre esta compensación. En Docker Desktop para Windows, el socket del host es \\.\pipe\docker_engine; la sintaxis de bind-mount anterior se traduce correctamente. El patrón es general; no se ha evaluado específicamente con el sandbox de skills de zeroclaw en este documento.

  • Límites de recursos. Docker Desktop en Windows asigna RAM/CPU mediante %USERPROFILE%\.wslconfig, que de forma predeterminada usa la mitad de la RAM del host. La superficie de configuración completa está documentada en la referencia Advanced settings configuration in WSL de Microsoft. Un punto de partida razonable para una implementación de ZeroClaw de un solo usuario es:

    [wsl2]
memory=8GB
processors=4

    Auméntalo si ejecutas cargas de trabajo intensivas de skills o inferencia local de LLM dentro de la misma distribución de WSL; esto es una guía de dimensionamiento, no un requisito estricto de la imagen.

Dependencias del sistema

Las compilaciones de Windows usan la cadena de herramientas MSVC. Para compilar desde el código fuente necesitas:

  • Herramientas de compilación de Visual Studio (o Visual Studio completo) con la carga de trabajo “Desarrollo de escritorio con C++”
  • Rust estable (vía rustup)

Si estás usando la Opción 1, no necesitas el toolchain de Rust; el binario es autónomo. La Opción 2 (setup.bat --prebuilt) está pensada para usar la misma ruta del binario, pero el script actual todavía verifica cargo antes de llegar a la rama prebuilt; consulta el problema conocido más arriba.

Ejecutándose como un servicio

En Windows, ZeroClaw se instala como una tarea programada de ámbito de usuario denominada ZeroClaw Daemon. No existe la opción de Servicio de Windows / LocalSystem en la versión actual; la ruta de código subyacente siempre instala una tarea programada, independientemente de si zeroclaw service install se ejecuta desde un shell con privilegios elevados o sin ellos.

zeroclaw service install
zeroclaw service start

Esto crea una tarea en el Programador de tareas (taskschd.msc) en tu cuenta de usuario que se inicia al iniciar sesión. Adminístrala mediante:

zeroclaw service status
zeroclaw service restart
zeroclaw service stop
zeroclaw service logs

Acerca de --service-init. La CLI expone un indicador --service-init [auto|systemd|openrc] para mantener la coherencia entre plataformas, pero en Windows no tiene efecto; siempre se utiliza la ruta de tareas programadas.

Los registros se guardan en %USERPROFILE%\.zeroclaw\logs\ (específicamente, en <config_dir>/logs/, donde <config_dir> tiene como valor predeterminado %USERPROFILE%\.zeroclaw\). Sin embargo, el wrapper de la tarea programada se ubica junto al archivo de configuración en %USERPROFILE%\.zeroclaw\zeroclaw-daemon.cmd. Solo los archivos de salida del daemon (daemon.stdout.log / daemon.stderr.log) se escriben en logs\.

Instalaciones de servidor / multiusuario. La compatibilidad nativa con Windows Service / LocalSystem está en la hoja de ruta, pero aún no se ha implementado. Por ahora, en un equipo servidor, instala ZeroClaw con la cuenta con la que debe ejecutarse el agente; la ruta de tarea programada lo iniciará al iniciar sesión ese usuario. Si necesitas que se inicie antes de que cualquier usuario inicie sesión, usa Task Scheduler → ZeroClaw Daemon → Properties → General → “Run whether user is logged on or not.”

Actualizar

Manual (ruta de la opción 1)

Vuelve a ejecutar el bloque de instalación de PowerShell de la Opción 1 con el nuevo $ver. El nuevo zip sobrescribe el zeroclaw.exe existente en su lugar. Luego:

zeroclaw service restart

setup.bat

Vuelve a descargar la última versión y vuelve a ejecutar setup.bat --prebuilt (o la bandera que usaste originalmente). Luego:

zeroclaw service restart

Scoop

scoop update zeroclaw       # subject to the staleness caveat above
zeroclaw service restart

Desde la fuente

cd C:\path\to\zeroclaw
git pull
cargo install --locked --path . --force
zeroclaw service restart

Desinstalar

Detenga y elimine la tarea programada:

zeroclaw service stop
zeroclaw service uninstall

Eliminar el binario:

:Opción 1 (precompilado manual) o setup.bat
rmdir /s /q "%USERPROFILE%\.zeroclaw\bin"

:: Option 3 (cargo install)
del "%USERPROFILE%\.cargo\bin\zeroclaw.exe"

:: Option 4 (Scoop)
scoop uninstall zeroclaw

Eliminar config, workspace y logs (opcional; esto elimina el historial de conversaciones):

rmdir /s /q "%USERPROFILE%\.zeroclaw"

La versión anterior de este documento hacía referencia a %LOCALAPPDATA%\ZeroClaw\; esa ruta no se utiliza en la versión actual; solo se usa %USERPROFILE%\.zeroclaw\.

Cosas a tener en cuenta

  • Rutas largas. Algunos sistemas de archivos de Windows aún limitan la longitud de las rutas a 260 caracteres. Habilita la compatibilidad con rutas largas si encuentras errores path too long durante una compilación desde el código fuente:

    reg add HKLM\SYSTEM\CurrentControlSet\Control\FileSystem /v LongPathsEnabled /t REG_DWORD /d 1 /f

  • SmartScreen. El binario sin firmar puede activar SmartScreen en el primer inicio desde el Explorador (doble clic). Clic derecho → Propiedades → “Desbloquear” es la solución alternativa estándar hasta que agreguemos un MSI firmado. Iniciar desde PowerShell o cmd.exe normalmente no activa SmartScreen.

  • Detención por inactividad/batería del Programador de tareas. De forma predeterminada, Windows puede finalizar las tareas programadas durante la inactividad o el uso de la batería. La tarea instalada ZeroClaw Daemon deshabilita estas condiciones, pero si la instalaste a través de una versión anterior, puedes verificarlo en Programador de tareas → ZeroClaw Daemon → Propiedades → Condiciones:

    • “Iniciar la tarea solo si el equipo está conectado a la corriente alterna (CA)”: sin marcar
    • “Detener si el equipo cambia a la alimentación por batería”: sin marcar
    • «Iniciar la tarea solo si el equipo lleva inactivo…»: sin marcar

  • Autenticación por contraseña de OpenSSH. Si controlas Windows a través de SSH y no se acepta la clave pública, coloca tu clave en C:\Users\<user>\.ssh\authorized_keys (usuario normal) o C:\ProgramData\ssh\administrators_authorized_keys (cuando inicies sesión como miembro de Administrators).
