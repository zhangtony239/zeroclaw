Protocolo del complemento

Este documento define el protocolo entre el host de plugins de ZeroClaw y los módulos de plugins WASM.

Estado de implementación

El puente de tiempo de ejecución actual es transitorio. crates/zeroclaw-plugins todavía admite el protocolo de herramientas de cadenas/JSON de la era de Extism documentado a continuación, de modo que los plugins prototipo existentes puedan seguir funcionando mientras se implementa el host WIT.

La arquitectura objetivo aceptada es el siguiente contrato: #6943 y FND-001 migran los plugins de ZeroClaw a componentes WASI definidos con WIT, compilados para wasm32-wasip2 y alojados directamente a través de wasmtime. Las interfaces WIT se encuentran en wit/v0/; consulta wit/VERSIONING.md para conocer las reglas de compatibilidad actuales.

Considera el nuevo trabajo de diseño de plugin-host como trabajo de WIT / wasm32-wasip2. Las adiciones al protocolo específico de Extism deben limitarse a correcciones de compatibilidad necesarias durante la migración.

Estructura del complemento

Un plugin es un directorio que contiene:

my-plugin/
  manifest.toml    # Plugin metadata and permissions
  plugin.wasm      # Compiled WASM module (optional for skill-only plugins)

Los complementos se descubren desde ~/.zeroclaw/plugins/ (configurable mediante plugins.plugins_dir en la configuración).

Estructura de plugin solo de habilidades (paquete markdown)

Un plugin cuya única capacidad es skill incluye skills en un directorio skills/ con formato agentskills.io y omite wasm_path:

my-toolkit/
  manifest.toml              # capabilities = ["skill"]
  README.md                  # descripción general opcional del paquete
  skills/
    design-review/
      SKILL.md
      scripts/
      references/
    code-review/
      SKILL.md
    data-analysis/
      SKILL.md
      references/

Cada SKILL.md debe incluir frontmatter YAML con los campos name y description; el runtime rechaza los paquetes cuyas skills omitan cualquiera de los dos en el momento del descubrimiento, en lugar de en la primera invocación. Las skills se registran con IDs con espacio de nombres de plugin con el formato plugin:<plugin-name>/<skill-name> (p. ej. plugin:my-toolkit/design-review) para evitar colisiones con las skills creadas por el usuario y entre paquetes.

Exportaciones actuales del puente Extism

Las exportaciones a continuación describen el puente Extism actual. Los componentes WIT usan las interfaces en wit/v0/.

tool_metadata

Firma: (String) -> String

Se llama una sola vez al cargar el plugin para recuperar los metadatos de la herramienta. La cadena de entrada se ignora (pasar una cadena vacía). Devuelve JSON:

{
  "nombre": "my_tool",
  "descripción": "Lo que hace esta herramienta",
  "esquema_de_parámetros": {
    "tipo": "objeto",
    "requerido": ["prompt"],
    "propiedades": {
      "prompt": {
        "tipo": "cadena",
        "descripción": "La entrada del prompt"
      }
    }
  }
}

El parameters_schema sigue el formato de JSON Schema y se presenta al LLM para la llamada de herramientas.

execute

Firma: (String) -> String

Se llama cada vez que se invoca la herramienta. La entrada es JSON que coincide con el parameters_schema. Devuelve JSON:

{
  "éxito": true,
  "salida": Texto de resultado mostrado al LLM,
  "error": null
}

En caso de fallo:

{
  "éxito": false,
  "salida": "",
  "error": Descripción de lo que salió mal
}

Funciones de host

Las funciones de host son proporcionadas por el runtime de ZeroClaw y pueden invocarse desde dentro del plugin WASM. Cada una está condicionada a un permiso del manifiesto; invocarla sin el permiso requerido devuelve un error.

zc_http_request

Permiso: http_client

Entrada: Cadena JSON

{
  "método": "POST",
  "url": "https://api.example.com/v1/generate",
  "encabezados": {
    "Autorización": "Token de portador123",
    "Content-Type": "application/json"
  },
  "body": "{\"prompt\": \"hola\"}"
}

Métodos admitidos: GET, POST, PUT, DELETE, PATCH, HEAD. Tiempo de espera: 120 segundos.

Salida: Cadena JSON

{
  "estado": 200,
  "body": "{\"result\": \"mundo\"}",
  "encabezados": {
    "tipo de contenido": "application/json"
  }
}

Configuración por plugin (__config)

Permiso: config_read

Un plugin no lee las variables de entorno del proceso. El host resuelve la sección de configuración propia del plugin desde [plugins.entries.<alias>] y la inyecta en la entrada de execute bajo la clave reservada __config:

{
  "prompt": un atardecer,
  __config: { "api_key": "...", "base_url": "..." }
}

Los operadores los establecen a través de la gramática de anulación schema-mirror, p. ej. ZEROCLAW_plugins__entries__<alias>__config__api_key=$FAL_KEY. Los valores son secretos y se cifran en reposo bajo el .secret_key adyacente. Un plugin solo ve su propia sección.

Escribir un plugin en Rust para el puente actual

Esta sección describe el puente actual de Extism.

Dependencias

La tabla [workspace] es necesaria para evitar que Cargo busque un espacio de trabajo padre.

Declaración de funciones de host

#![allow(unused)]
fn main() {
use extism_pdk::*;

#[host_fn]
extern "ExtismHost" {
    fn zc_http_request(input: String) -> String;
}
}

Llámalos con unsafe { zc_http_request(json_string)? }. La configuración por plugin llega en la entrada de execute bajo __config; léela desde la entrada parseada en lugar de hacerlo mediante una llamada al host.

Implementando exportaciones

#![allow(unused)]
fn main() {
#[plugin_fn]
pub fn tool_metadata(_input: String) -> FnResult<String> {
    Ok(serde_json::to_string(&serde_json::json!({
        "nombre": "my_tool",
        "descripción": "Hace algo útil",
        "esquema_de_parámetros": {
            "tipo": "objeto",
            "requerido": ["entrada"],
            "propiedades": {
                "entrada": { "tipo": "cadena" }
            }
        }
    }))?)
}

#[plugin_fn]
pub fn execute(input: String) -> FnResult<String> {
    let args: serde_json::Value = serde_json::from_str(&input)?;
    let input_val = args["entrada"].as_str().unwrap_or("");

    // ... realizar el trabajo, llamar a las funciones del host según sea necesario ...

    Ok(serde_json::to_string(&serde_json::json!({
        "éxito": true,
        "salida": format!("Procesado: {input_val}")
    }))?)
}
}

Construyendo

Para el bridge actual de Extism, compila el plugin como un módulo WASI Preview 1:

sh

# Instalar el objetivo WASM (una sola vez)
rustup target add wasm32-wasip1

# Construir
cargo build --target wasm32-wasip1 --release

El archivo .wasm de salida se encuentra en target/wasm32-wasip1/release/<crate_name>.wasm. Copia este archivo junto a tu manifest.toml.

Para el target aceptado WIT / Component Model, compila los componentes de plugin para wasm32-wasip2 y usa wit/v0/ como fuente de la interfaz.

Instalando

sh

# Copiar al directorio del complemento
zeroclaw plugin install /path/to/my-plugin/

# O manualmente
cp -r my-plugin/ ~/.zeroclaw/plugins/my-plugin/

Configuración

Habilita el sistema de plugins mediante las secciones plugins y plugins.security (gateway, zerocode o zeroclaw config set): consulta la referencia de configuración para ver todos los campos, los valores predeterminados y el enum signature_mode.

La característica plugins-wasm habilita la integración actual del entorno de ejecución de plugins. La ruta directa del host wasmtime usa plugins-wasm-runtime-only, plugins-wasm-cranelift y plugins-wasm-pulley para elegir la estrategia de ejecución más pequeña que coincida con la plataforma de destino.
