ZeroClaw Docs

Windows

在 Windows 10 / 11 上安装、更新、作为 Windows 计划任务运行以及卸载。

如果你运行的是 WSL2，可以改为参照 Linux 安装；install.sh 在 WSL 下无需修改即可运行。

关于 setup.bat 的说明。 #6118 硬性中止故障（32 位 set /a 磁盘空间溢出以及未转义括号导致的 if/else 解析错误）已在 #6137 中修复，并随 v0.7.4 及更高版本发布。当前版本可正常完成。仍存在一个注意事项：setup.bat --prebuilt 在进入预构建分支之前仍会检查 cargo，因此手动预构建路径（下方的选项 1）才是真正无需 Rust 的安装方式。从源代码构建（选项 3）同样可行。

安装

选项 1：预构建二进制文件（推荐）

下载最新的 Windows 发行版 zip 包，解压 zeroclaw.exe，然后将其添加到你的 PATH。

在 PowerShell 提示符中：

# 幂等性：当 zeroclaw 已安装最新版本且位于用户 PATH 中时，
# 重新运行此代码块不会产生任何影响。当某次发布
# 提升版本号后，版本检查会失败，安装部分将再次运行。
$ver = (Invoke-RestMethod 'https://api.github.com/repos/zeroclaw-labs/zeroclaw/releases/latest').tag_name.TrimStart('v')
$dst = "$env:USERPROFILE\.zeroclaw\bin"
$exe = "$dst\zeroclaw.exe"

$current = if (Test-Path $exe) {
    ((& $exe --version 2>$null) | Select-String -Pattern '\d+\.\d+\.\d+').Matches.Value
} else { '' }

if ($current -ne $ver) {
    $url = "https://github.com/zeroclaw-labs/zeroclaw/releases/download/v$ver/zeroclaw-x86_64-pc-windows-msvc.zip"
    New-Item -ItemType Directory -Force -Path $dst | Out-Null
    Invoke-WebRequest -Uri $url -OutFile "$env:TEMP\zeroclaw.zip" -UseBasicParsing
    Expand-Archive -Force -Path "$env:TEMP\zeroclaw.zip" -DestinationPath $dst
}

$userPath = [Environment]::GetEnvironmentVariable('Path', 'User')
if (($userPath -split ';') -notcontains $dst) {
    [Environment]::SetEnvironmentVariable('Path', "$dst;$userPath", 'User')
}
if (($env:Path -split ';') -notcontains $dst) {
    $env:Path = "$dst;$env:Path"
}

& $exe quickstart

该 zip 包提供一个独立的二进制文件，无需 Rust 工具链，也无需 Visual Studio Build Tools。

安装后，请验证：

zeroclaw --version    # matches the latest release

选项 2：setup.bat（来自发行版）

setup.bat --prebuilt

标志：

标志行为
--prebuilt从 GitHub Releases 下载预编译的二进制文件（到达后速度最快；当前脚本仍会优先检查 cargo）
--minimal仅构建核心（无通道、无硬件）
--standard使用常用渠道构建（Telegram、Discord、Slack、Matrix）
--full构建所有内容

⚠️ 已知问题（当前）。 setup.bat --prebuilt 在进入预构建分支之前仍会检查 cargo，因此选项 2 无法兑现无需 Rust 的承诺。如果你没有安装 Rust 工具链，请使用上面的选项 1。

历史问题（v0.7.4 之前）。 早期版本存在两个硬性中止故障以及一个引导命令不匹配的问题，已在 #6118 中报告：一个 32 位 set /a 磁盘空间溢出问题（Invalid number. Numbers are limited to 32-bits of precision.），一个未转义括号的 if/else 解析错误（.[0m was unexpected at this time.），以及一个最终的 zeroclaw init 提示。这些问题均已在 #6137（v0.7.4 / 0.7.5 / 0.8.0）中修复；当前版本会打印 zeroclaw quickstart 并正常完成。

选项 3：从源代码

需要 Rust（rustup）和 Visual Studio 构建工具：

git clone https://github.com/zeroclaw-labs/zeroclaw
cd zeroclaw
cargo install --locked --path .
zeroclaw quickstart

选项 4：Scoop（目前已过时）

⚠️ 仓库中的 Scoop manifest 已固定到 v0.5.9 版本（落后 master 23 个补丁版本）。在发布时的 CI 钩子更新它之前，建议使用方案 1 或方案 3。如果你确实要使用 Scoop 并遇到问题，请针对 dist/scoop/zeroclaw.json 提交一个 PR。

scoop install zeroclaw     # currently installs an older release; see warning above
zeroclaw quickstart

选项 5：Docker

ZeroClaw 在 ghcr.io/zeroclaw-labs/zeroclaw:latest 发布 Linux 容器镜像（带标签的发行版则使用 :vX.Y.Z）。在 Windows 上，可通过 Docker Desktop 或在 WSL 发行版中使用 sudo apt install docker.io 来运行；两种方式都可行，且容器行为完全一致。

快速开始：

# 用于配置和工作区的持久化卷；容器内 ZeroClaw 的数据目录为 /zeroclaw-data
docker run -d --name zeroclaw `
  --restart=unless-stopped `
  -p 42617:42617 `
  -v zeroclaw-data:/zeroclaw-data `
  ghcr.io/zeroclaw-labs/zeroclaw:latest

# 查看首次运行的日志
docker logs -f zeroclaw

# 健康检查（无需身份验证）
curl http://localhost:42617/health

# （可选）配对客户端。注意：已发布的镜像默认为
# `require_pairing = false`，因此默认情况下不会生成配对码，
# 且 `/api/*` 接受未经身份验证的请求。要启用配对，
# 请覆盖配置（在容器内的
# /zeroclaw-data/.zeroclaw/config.toml 中设置 `require_pairing = true`）并重启；
# 随后首次启动时会在 stdout 打印一个一次性配对码，
# 之后客户端将其 POST 到 `/pair`：
curl -X POST http://localhost:42617/pair -H 'X-Pairing-Code: <code-from-logs>'

镜像信息（已对照 ghcr.io/zeroclaw-labs/zeroclaw:latest 验证）：

  • 基础镜像： gcr.io/distroless/cc-debian13:nonroot（发布阶段；dev 阶段为 debian:trixie-slim）
  • ENTRYPOINT ["zeroclaw"]、CMD ["daemon"]：不带参数运行时启动守护进程和网关
  • EXPOSE 42617：守护进程和网关都监听此端口
  • 数据目录： /zeroclaw-data（配置：/zeroclaw-data/.zeroclaw/config.toml，工作区：/zeroclaw-data/workspace）。在此挂载命名卷或绑定挂载以实现持久化；请注意这不是 /root/.zeroclaw。
  • 配对： 发布的镜像默认设置 require_pairing = false，因此 /api/* 开箱即用地接受无需认证的请求。启用配对后（在 /zeroclaw-data/.zeroclaw/config.toml 中设置 require_pairing = true 并重启），守护进程会在首次启动时将一次性代码打印到 stdout，随后客户端需通过带有 X-Pairing-Code 标头的请求将其 POST 到 /pair，之后任何认证端点才会响应。
  • Web 仪表板：在发布镜像中默认打包并提供服务。镜像设置了 gateway.web_dist_dir = "/usr/share/zeroclawlabs/web/dist"，并在该位置包含了构建好的前端，因此网关开箱即用地提供 SPA 回退服务。这些资源位于 /zeroclaw-data 挂载点之外，因此 -v …:/zeroclaw-data 卷挂载无法遮蔽它们（参见 #6400）。

针对捆绑的 Dockerfile 从源代码构建：

git clone https://github.com/zeroclaw-labs/zeroclaw
cd zeroclaw
docker build -t zeroclaw:local -f Dockerfile.debian .

已在 Windows + Docker 上验证：

  • 容器行为与 Linux 一致。 已在 Windows 11 build 26200.8313 上的 WSL Debian 中拉取并运行 ghcr.io/zeroclaw-labs/zeroclaw:latest。镜像正常启动，网关在 :42617 上监听，/health 返回有效的 JSON。在配置中设置 require_pairing = true 并重启容器后，/pair 上的配对码流程也能按文档所述正常工作。
  • 无需 Docker Desktop 的 Docker。 运行 wsl --install 启用 WSL2，然后在 WSL 发行版中执行 sudo apt install docker.io，即可直接获得守护进程；已验证可拉取并无需修改即可运行已发布的镜像。

主机端最佳实践：通用的 Docker + WSL2 指导，并非 zeroclaw 特有的运行时声明。在适用情况下，内容来源于 Microsoft Learn 和 Docker 的官方文档：

  • 卷挂载。 将 Windows 侧路径绑定挂载（-v C:/Users/...:/zeroclaw-data）到 Linux 容器会跨越 WSL2 ⇄ Windows 文件系统边界；Microsoft 在 WSL 文件系统参考文档中记录了相关布局及跨操作系统路径的影响。为获得接近原生的性能，建议优先使用 Docker 命名卷（-v zeroclaw-data:/zeroclaw-data），或将工作区存储在 WSL 文件系统内（\\wsl$\Debian\home\...）。

  • 网络。 默认的 WSL2 网络采用 NAT，因此在 -p 端口转发后，可以通过 localhost:<port> 从 Windows 访问容器中的服务（已在 Windows 11 + WSL2 上验证）。如果需要从局域网中的其他主机访问容器，或运行依赖容器间 DNS 的多容器配置，请参照 Microsoft 的 Mirrored mode networking 文档切换到镜像模式，方法是在 %USERPROFILE%\.wslconfig 中添加：

    [wsl2]
networkingMode=mirrored

  • 在 Docker 下使用守护进程，而非任务计划程序。 容器内部没有 Windows 任务计划程序。请使用 Docker 的重启策略，如上面示例中的 --restart=unless-stopped，以实现守护进程模式启动。发布的镜像以 PID 1 / 非 root 用户身份运行；容器_即_服务；请勿在容器内运行 zeroclaw service install。

  • 通过主机 Docker 套接字实现技能沙盒。 ZeroClaw 的技能执行沙盒可以调用 Docker。如果你将 ZeroClaw 本身运行在容器中，并希望技能沙盒也使用 Docker，请挂载主机的 Docker 套接字，使子容器运行在主机守护进程上，而不是嵌套使用 Docker-in-Docker：

    # PowerShell / cmd.exe：使用单个前导斜杠。
# Git Bash / MINGW：使用 //var/run/docker.sock 以绕过 MSYS 路径重写。
-v /var/run/docker.sock:/var/run/docker.sock

    请注意，挂载 Docker socket 会向容器内的任何内容授予等同于 root 的主机访问权限；Docker 的保护 Docker 守护进程 socket 页面介绍了其中的权衡取舍。在 Docker Desktop for Windows 上，主机 socket 为 \\.\pipe\docker_engine；上述绑定挂载语法可以正确转换。该模式具有通用性；本文档中并未专门针对 zeroclaw 的技能沙箱进行基准测试。

  • 资源限制。 Windows 上的 Docker Desktop 通过 %USERPROFILE%\.wslconfig 分配 RAM/CPU，默认值为主机 RAM 的一半。完整的配置选项记录在 Microsoft 的 WSL 中的高级设置配置 参考文档中。对于单用户 ZeroClaw 部署，一个合理的初始配置范围是：

    [wsl2]
memory=8GB
processors=4

    如果你在同一个 WSL 发行版中运行繁重的技能工作负载或本地 LLM 推理，可以适当调高；这只是容量规划建议，并非镜像的硬性要求。

系统依赖

Windows 版本使用 MSVC 工具链。要从源代码构建，您需要：

  • Visual Studio Build Tools（或完整的 Visual Studio），并安装“使用 C++ 的桌面开发”工作负载
  • Rust 稳定版（通过 rustup）

如果你使用的是选项 1，则无需 Rust 工具链；该二进制文件是自包含的。选项 2（setup.bat --prebuilt）旨在使用相同的二进制文件路径，但当前脚本在执行到预构建分支之前仍会检查 cargo；请参阅上面的已知问题。

作为服务运行

在 Windows 上，ZeroClaw 会以名为 ZeroClaw Daemon 的用户范围计划任务进行安装。当前版本不提供 Windows Service / LocalSystem 选项；无论 zeroclaw service install 是在提升权限的 shell 还是非提升权限的 shell 中运行，底层代码路径始终会安装一个计划任务。

zeroclaw service install
zeroclaw service start

这会在任务计划程序（taskschd.msc）中以您的用户账户创建一个在登录时启动的任务。可通过以下方式管理：

zeroclaw service status
zeroclaw service restart
zeroclaw service stop
zeroclaw service logs

关于 --service-init。 CLI 提供了 --service-init [auto|systemd|openrc] 标志以保持跨平台一致性，但在 Windows 上该标志为空操作；始终使用计划任务方式。

日志会写入 %USERPROFILE%\.zeroclaw\logs\（具体来说是 <config_dir>/logs/，其中 <config_dir> 默认为 %USERPROFILE%\.zeroclaw\）。不过，计划任务包装脚本本身位于配置文件旁边的 %USERPROFILE%\.zeroclaw\zeroclaw-daemon.cmd。只有守护进程的输出文件（daemon.stdout.log / daemon.stderr.log）才会写入 logs\ 目录下。

服务器/多用户安装。 原生 Windows Service / LocalSystem 支持已列入路线图，但尚未实现。目前，在服务器上，请在代理应当运行所用的账户下安装 ZeroClaw；计划任务路径会在该用户登录时启动它。如果你需要它在任何用户登录之前启动，请使用 任务计划程序 → ZeroClaw Daemon → 属性 → 常规 → “不管用户是否登录都要运行”。

更新

手动（选项 1 路径）

使用新的 $ver 重新运行选项 1 中的 PowerShell 安装代码块。新的 zip 文件会就地覆盖现有的 zeroclaw.exe。然后：

zeroclaw service restart

setup.bat

重新下载最新版本并重新运行 setup.bat --prebuilt（或你最初使用的其他标志）。然后：

zeroclaw service restart

Scoop

scoop update zeroclaw       # subject to the staleness caveat above
zeroclaw service restart

从源代码

cd C:\path\to\zeroclaw
git pull
cargo install --locked --path . --force
zeroclaw service restart

卸载

停止并删除计划任务：

zeroclaw service stop
zeroclaw service uninstall

移除二进制文件：

: 选项 1（手动预构建）或 setup.bat
rmdir /s /q "%USERPROFILE%\.zeroclaw\bin"

:: Option 3 (cargo install)
del "%USERPROFILE%\.cargo\bin\zeroclaw.exe"

:: Option 4 (Scoop)
scoop uninstall zeroclaw

移除配置、工作区和日志（可选；此操作将删除对话历史记录）：

rmdir /s /q "%USERPROFILE%\.zeroclaw"

此文档的早期版本提到了 %LOCALAPPDATA%\ZeroClaw\；当前版本不使用该路径，仅使用 %USERPROFILE%\.zeroclaw\。

注意事项

  • 长路径。 某些 Windows 文件系统仍将路径长度限制为 260 个字符。如果在源码构建过程中遇到 path too long 错误，请启用长路径支持：

    reg add HKLM\SYSTEM\CurrentControlSet\Control\FileSystem /v LongPathsEnabled /t REG_DWORD /d 1 /f

  • SmartScreen。 未签名的二进制文件在首次从资源管理器启动（双击）时可能会触发 SmartScreen。右键单击 → 属性 → “解除锁定”是标准的解决方法，直到我们添加经过签名的 MSI 为止。从 PowerShell 或 cmd.exe 启动通常不会触发 SmartScreen。

  • 任务计划程序的空闲/电池停止设置。 默认情况下，Windows 可能会在系统空闲或使用电池时终止计划任务。已安装的 ZeroClaw Daemon 任务会禁用这些条件，但如果你是通过较旧的版本安装的，可以在 任务计划程序 → ZeroClaw Daemon → 属性 → 条件 下进行验证：

    • “仅在计算机使用交流电源时启动任务”：未选中
    • “切换到电池供电时停止”：未勾选
    • “仅当计算机空闲达到…时才启动任务”：未勾选

  • OpenSSH 密码认证。 如果你通过 SSH 操作 Windows 但公钥未被接受，请将你的密钥放入 C:\Users\<user>\.ssh\authorized_keys（普通用户）或 C:\ProgramData\ssh\administrators_authorized_keys（以 Administrators 组成员身份登录时）。
