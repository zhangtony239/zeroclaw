ZeroClaw Docs

提供程序目录

ZeroClaw 附带的每个模型提供商系列。针对每一项：配置结构、关于身份验证和端点行为的说明，以及在 [providers.models.<type>.<alias>] 下使用的槽位键。

有关通用字段（api_key、uri、model 等）及其解析顺序，请参阅配置。

以下示例使用 home 作为别名，以强调别名部分是由操作员自行选择的，可以使用任何合适的名称（work、personal、cn、prod……）。在 agent 中通过 model_provider = "<type>.<alias>" 来引用它。

原生

Anthropic：插槽 anthropic

支持来自 Claude Pro/Team 订阅的 OAuth 令牌（sk-ant-oat*），无需单独的 API 计费。支持流式传输、工具调用、视觉和推理。自定义端点（与 Anthropic 兼容的代理，例如 Z.AI 的 Anthropic API）也使用此插槽：设置 uri 即可覆盖。

OpenAI：插槽 openai

GPT-4o、GPT-5、o 系列推理模型。推理令牌以 ReasoningDelta 事件形式呈现；详见 流式传输。

OpenAI Codex：启用 requires_openai_auth = true 的 openai 槽位

OpenAI Codex 订阅认证位于 openai 槽位中。设置 wire_api = "responses" 可将请求路由至 POST /v1/responses，设置 requires_openai_auth = true 可使用 Codex 订阅登录（来自 Codex CLI 自身的 ~/.codex/auth.json），而非条目中的 api_key 字段。订阅路径不会读取 OPENAI_API_KEY；该变量仅适用于按量计费的 openai API 密钥模式。有关凭据模型，请参阅 Provider Configuration → OAuth and subscription auth。

Ollama：插槽 ollama

通过 Ollama 原生 /api/chat 进行本地推理。通过 format 实现基于 Schema 的结构化输出。无需 API 密钥。

Bedrock：插槽 bedrock

Gemini：插槽 gemini

Google 的 Gemini API。支持视觉和预执行的接地搜索（有关 PreExecutedToolCall 事件，请参阅 流式传输）。

Gemini CLI：插槽 gemini_cli

通过 shell 调用 gemini CLI；使用该 CLI 现有的身份验证。

Azure OpenAI：插槽 azure

resource、deployment 和 api_version 存放在此类型化配置中，它们不会从环境变量读取。

Copilot：插槽 copilot

使用 GitHub Copilot 订阅进行 agent 推理。身份验证使用从 GitHub 获取的 Copilot OAuth 令牌。

Telnyx：插槽 telnyx

面向语音的 AI 端点。与 clawdtalk 通道配对，用于实时 SIP 通话。

KiloCLI：插槽 kilocli

通过 KiloCLI 进行本地推理。

Kilo AI Gateway：插槽 kilo

[providers.models.kilo.home]
model   = "anthropic/claude-sonnet-4-6"
api_key = "..."
# endpoint = "gateway"  # 默认 → https://api.kilo.ai/api/gateway

通过 Kilo AI Gateway 访问的 Cloud API。采用 Bearer-token 鉴权，提供多个模型层级（free、balanced、pro）。/models 端点是公开的（PUBLIC_MODEL_LISTING），因此无需凭据即可列出模型。由于它是实时查询的，它也是将定价信息带入成本费率编辑器的数据来源。共享的 models.dev 目录（kilo 键）仅在实时端点不可达时作为后备，且不包含定价信息。

命名迁移： kilo 现在指代此网关提供程序。KiloCLI 子进程提供程序保留其 kilocli 槽位（同义词 kilo-cli）。如果你之前在 kilo 简写下配置了 CLI 提供程序，请改用 kilocli。

所有插槽

每个规范槽位、其默认端点、是否在本地运行，以及由提供方注册表和配置模式生成的完整配置字段集。点击槽位可展开其字段；点击字段可查看如何设置。没有固定默认值的槽位需要在别名条目上设置 uri（Azure、custom、多区域系列、CLI shim）。

共享字段

每个提供程序槽位都接受这些字段。各提供程序特有的额外字段在下方分别列出。

api_key 🔑 secret · default —

此 model_provider 的密钥 API 令牌。请从 model_provider 的控制台获取（OpenAI platform、Anthropic console、OpenRouter keys 页面等）。尽可能通过操作系统密钥环存储；切勿直接将其提交到 config.toml 中。

将它放置在任何表面上：

网关仪表板

打开 /config/providers.models/openrouter 并设置 providers.models.openrouter.<alias>.api_key 字段。

zerocode

在 Config 窗格中，设置 providers.models.openrouter.<alias>.api_key 字段。

zeroclaw config

zeroclaw config set providers.models.openrouter.<alias>.api_key    # 掩码输入，加密存储

环境变量

导出此覆盖配置（POSIX shell；可放入 ~/.bashrc、~/.zshrc、.env 或 Dockerfile 中）。将 <alias> 替换为实际的别名：

export ZEROCLAW_providers__models__openrouter__<alias>__api_key=

fallback string[] · default —

当此别名上的每个模型都失败后，按顺序尝试的其他提供方别名列表。每个条目都是指向 providers.models 的点分 <type>.<alias> 引用，并使用其自身的凭据、端点和模型进行解析。回退别名绝不会继承此别名的密钥。遍历采用深度优先方式：先用尽此别名的模型，然后依次深入每个回退别名（应用其自身的 fallback_models 和 fallback）。为空表示没有提供方级别的回退。

fallback_models string[] · default —

在转向 fallback 别名之前，按顺序尝试当前提供方上的备用模型。使用与主 model 相同的端点、密钥和请求头，仅更改模型标识符。当某个提供方提供了备用模型（例如更小或更旧的变体），且应在彻底切换提供方之前先行尝试时，请使用此选项。留空则仅尝试 model。

实战示例：Morph、GitHub Models、Upstage、Featherless、Arcee、Lambda AI、Inception

以下每一项都是标准的 OpenAI 兼容槽位：设置 model 和 api_key，并省略 uri（由所选类型的端点自动提供）。它们均不附带公开的模型索引，因此在你粘贴凭据之前，模型选择器将保持为空。一旦设置了密钥，ZeroClaw 就会从该提供商的实时 /models 端点列出模型。下方的模型 ID 仅作示例，请在供应商的控制台中确认当前的目录。

Morph：插槽 morph。快速应用编辑模型（morph-v3-large、morph-v3-fast 或 auto）。密钥来自 Morph dashboard。

GitHub Models：插槽 github_models（别名 github-models）。通过单个 GitHub 个人访问令牌即可使用 OpenAI / Meta / Microsoft 模型。创建一个具有 models 权限的 PAT（细粒度）；Copilot 令牌与此_并非_同一种凭据。模型 ID 带有发布者前缀（例如 openai/gpt-4o）。

Upstage：插槽 upstage。Solar Pro / Solar Mini（例如 solar-pro2）。密钥从 Upstage 控制台获取。

Featherless：插槽 featherless。无服务器开放权重模型，通过其 Hugging Face 仓库 ID 进行寻址（例如 meta-llama/Meta-Llama-3.1-8B-Instruct）。密钥来自 featherless.ai。

Arcee：插槽 arcee。原生模型包括 conductor、maestro、virtuoso-large、coder-large 和 blitz。密钥来自 Arcee 平台。Arcee 的平台 API 使用非标准的 /api/v1 基础路径；类型化端点已经考虑到这一点，因此仍然保持不设置 uri。

Lambda AI：插槽 lambda_ai（别名 lambda-ai）。Lambda 托管推理服务（例如 hermes3-405b）。密钥可从 Lambda Cloud 的 API 密钥页面获取。

Inception：插槽 inception。Mercury 扩散式 LLM 系列（mercury-coder 以及更新的 mercury-2）。密钥来自 Inception 平台。

凭据仅来自配置（api_key）或运行时的 --credential 覆盖，这些位置不会读取各提供商专属的 *_API_KEY 环境变量。

NEAR AI Cloud 示例：

[providers.models.nearai.tee]
model   = "..."       # 从 https://cloud-api.near.ai/v1/model/list 选择一个 modelId
api_key = "..."

nearai 槽位默认使用 https://cloud-api.near.ai/v1 并发送 Authorization: Bearer <api_key>。要将现有的 NEARAI_API_KEY shell 变量桥接到 ZeroClaw 的 schema-mirror 环境变量层，请设置 ZEROCLAW_providers__models__nearai__tee__api_key="$NEARAI_API_KEY"。

多家中国厂商提供不同区域的独立端点，且默认模型各不相同。请使用一个规范化的配置槽，并通过别名条目上类型化的 endpoint 字段来选择区域。

对于 Z.AI 的 Anthropic 兼容 API，请改用 [providers.models.anthropic.zai] 并设置 uri = "https://api.z.ai/api/anthropic"。

其余中文区域插槽（yi、hunyuan、qianfan、baichuan）显示在上方的全插槽表中；请通过别名条目上带类型的 endpoint 字段选择区域。

路由层

OpenRouter 被视为单一的一等提供方，而非元路由器。运行时只看到一个端点；OpenRouter 在该端点背后处理供应商分发。

对于按任务路由，可运行多个 agent 并让 channels 选择由哪个 agent 处理哪些流量，详见 Routing。如需更精细的配置内提示机制，请使用 [[model_routes]]。

缺少某些内容？

  • 如果端点兼容 OpenAI，请使用 custom 插槽并设置 uri。
  • 如果它在上方有自己的规范槽位，请使用该槽位，即使你只看到它的某个区域，该槽位的 endpoint 枚举也涵盖了其余部分。
  • 如果它使用非 OpenAI 的传输格式并需要自己的实现，请参阅自定义提供程序。
