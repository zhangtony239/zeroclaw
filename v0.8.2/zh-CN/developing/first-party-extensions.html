ZeroClaw Docs

第一方扩展架构

在添加或修改内置 ZeroClaw 提供方、通道、工具、内存后端或硬件/外设集成时，请使用此页面。

本页面不适用于进程外插件。如果扩展可以在核心二进制文件之外运行，请先考虑使用 Plugin protocol 或 MCP，而不是添加第一方实现。

选择内置或外部

在添加或删除代码之前，先为每项功能选择最小化且持久的归属位置，以保持核心二进制文件的精简。外部集成本身并不构成删除第一方工具的理由；其边界取决于 ZeroClaw 必须为运维人员、智能体和现有配置所保留的契约。

Surface使用场景避免使用场景
基线内置能力该能力是 agent 基线操作约定的一部分，需要紧密集成自治/回执机制，或必须在配置 plugins、skills 或 MCP servers 之前即可工作。该行为只是对某个供应商 API、产品、SaaS 或可选本地程序的简单封装。
功能门控的第一方实现该实现属于第一方代码、涉及安全敏感内容，或需要 Rust trait 集成，但其依赖项、平台或二进制体积成本不应影响最小化构建。新建 crate 仍需通过架构图、RFC 和稳定性层级检查。该功能标志会成为隐藏的兼容性陷阱，或者成为配置/状态的第二真实来源。
WASM 插件该功能应当能够独立安装、升级、授权或分发，同时仍使用 ZeroClaw 的插件 ABI 和清单权限。当前插件 ABI 无法表达所需的安全、生命周期或数据契约。
技能包这个价值主要体现在指令、提示词、可复用工作流、脚本或可通过现有工具运行的 CLI 配方上。该能力需要新的可信运行时原语、长期运行的服务，或对配置/Schema 的直接所有权。
MCP 服务器本地或远程服务已暴露适当的 Model Context Protocol 接口，或者该集成应保留在 ZeroClaw 进程之外。该项目需要稳定的第一方保证、离线默认行为，或服务器无法提供的 ZeroClaw 专属安全模型。
由 CLI 提供支持的集成成熟的外部 CLI 已经负责处理身份验证、API 漂移和平台行为，ZeroClaw 只需通过现有的 shell/工具策略调用它即可。CLI 输出过于不稳定、需要宽泛的环境权限，或会隐藏工具回执和审批策略中的副作用。

用此作为决策清单：先记录所需的契约，然后在需要代码验证时选择一个小型迁移候选项。在兼容性、配置、安全策略、运维可见性和回滚机制明确之前，不要先移除工具或集成。

选择最小的曲面

在编写代码之前，先确定哪个扩展层负责该行为。

工作首站通常不变
OpenAI 兼容模型端点自定义提供方zeroclaw-runtime
新模型传输格式或提供商系列crates/zeroclaw-api/src/model_provider.rs、crates/zeroclaw-providers/、crates/zeroclaw-config/频道、工具
新消息平台crates/zeroclaw-api/src/channel.rs、crates/zeroclaw-channels/、crates/zeroclaw-config/提供商路由
新的代理可调用功能crates/zeroclaw-api/src/tool.rs、crates/zeroclaw-tools/、风险策略文档通道适配器
新的内存后端crates/zeroclaw-api/src/memory_traits.rs、crates/zeroclaw-memory/提供商或频道代码
新硬件/外设界面crates/zeroclaw-api/src/peripherals_traits.rs、crates/zeroclaw-hardware/运行时智能体循环

如果该变更需要新的配置、安全策略、生成的文档或运行时路由，那么它就不只是一个本地适配器。在实现之前，请查看架构图和 RFC 流程。

事实来源

不要在配置、运行时句柄、生成的文档和 UI 状态之间重复存储同一状态。在添加字段或缓存之前，请先明确单一可信来源。

数据事实来源
服务提供商系列槽位、别名、端点默认值、认证字段crates/zeroclaw-config/src/providers.rs 和类型化的提供程序配置结构体
通道别名、密钥、允许的对等节点、通道本地配置crates/zeroclaw-config/src/schema.rs 与对等组/IAM 配置
工具规范和模型可见的描述Tool 实现的 spec()、description() 和 schema；诸如 crates/zeroclaw-runtime/locales/en/tools.ftl 之类的 Fluent 目录在工具通过该路径接入时持有本地化的工具文本
仪表板使用的 Gateway API 模式zeroclaw_gateway::openapi::build_spec()
生成的配置文档配置模式生成器；请勿手动编辑生成的输出
提供方、通道、工具、内存和外设特征契约crates/zeroclaw-api/src/model_provider.rs、channel.rs、tool.rs、memory_traits.rs 和 peripherals_traits.rs
归因与日志记录约定crates/zeroclaw-api/src/attribution.rs 和日志架构
运行时回合执行crates/zeroclaw-runtime/src/agent/turn/

在需要时，有效的模式会从源头解析：传入 &Config、通过工厂解析、使用闭包捕获实时配置句柄，或者构建按调用物化的视图。当规范数据存放在配置中时，应避免在通道句柄内存储长期存在的快照，例如 allowed_users: Vec<String>。

共享实现清单

在添加新的第一方扩展或对现有扩展进行重大扩充时，请使用此核对清单。

  1. 先阅读类似的实现。 在创造新模式之前，先参照最接近的现有 provider、channel、tool、内存后端或硬件模块。
  2. 实现公共 trait。 将扩展保持在相关的 zeroclaw-api trait 之后。不要仅仅为了对某个集成进行特殊处理而修补运行时循环。
  3. 通过工厂注册。 当更改引入新实现时，请在现有的工厂边界处接入该实现。仅当依赖项或平台层面确有必要时，才添加功能开关。
  4. 在 schema 边界处添加配置。 当配置键发生变更时，应在 schema/源代码层定义默认值、兼容性行为、环境变量预期以及迁移影响。
  5. 将安全控制保持在正确的边界处。 通道在消息到达运行时之前，会强制执行通道专属的身份验证、对等组/允许列表以及配对策略。工具则需经过自主性/风险策略和回执处理。提供方不得将工具或凭据策略偷偷塞入模型 I/O 代码中。
  6. 实现来源标记和日志记录。 新的第一方集成应通过项目日志接口发出日志，并使用正确的 kind/alias 来源标记，而不是临时的追踪字段。
  7. 本地化面向用户的字符串。 CLI 消息、工具说明、引导提示和 UI 文本使用 Fluent 或所属的 UI 目录。日志和追踪信息保持稳定的英文。
  8. 记录操作员接口。 如果扩展新增了配置、命令、安装步骤、安全行为或兼容性要求，请更新相应的文档。
  9. 测试工厂和边界情况。 在适用时覆盖工厂注册，以及配置解析/默认值、认证或策略拒绝路径、错误处理，以及至少一条成功路径。
  10. 保持 PR 范围聚焦。 不要将适配器、运行时重构、配置迁移和文档重组捆绑在一起，除非该变更必须同时包含所有这些内容才能正常工作。

特定平台说明

提供者

当端点使用受支持的传输协议时，优先选择 custom 插槽或现有的 OpenAI 兼容系列。仅当端点需要新的系列、认证模型、能力声明或传输转换时，才添加第一方提供方。

提供者 PR 通常涉及：

  • crates/zeroclaw-config/src/providers.rs 和类型化配置结构体；
  • crates/zeroclaw-providers/src/
  • 提供商目录/配置文档；
  • 工厂路由、别名、端点默认值、能力标志和错误处理的测试。

频道

通道是用户可见的信任边界。请将平台解码、去重、回复目标构建、发送者授权以及任何特定于通道的配对操作保留在通道边缘处理。不要让未经身份验证或未经授权的事件进入运行时后，再依赖运行时来对其进行甄别处理。

通道 PR 通常涉及：

  • crates/zeroclaw-api/src/channel.rs 仅在共享通道契约确实发生变化时；
  • crates/zeroclaw-channels/src/;
  • 通道配置架构和功能标志；
  • 通道文档，包括设置和安全说明；
  • 入站过滤、出站格式化、重试/错误行为以及工厂接线的测试。

工具

工具是智能体的操作，因此它们必须符合自主性模型。工具的 PR 应当定义该工具的风险、参数、结果结构、回执行为以及面向用户的描述。

工具 PR 通常涉及：

  • crates/zeroclaw-tools/src/
  • crates/zeroclaw-runtime/locales/en/tools.ftl
  • 当风险行为发生变化时，请参阅安全/自治文档；
  • 针对 schema、允许/拒绝行为、成功、失败以及面向回执的输出进行测试。

内存后端

内存后端必须保留 agent/session 范围限定。不要绕过 agent 范围限定的封装器，也不要因为某个后端实现起来更简单就存储未限定范围的数据。

一个内存 PR 通常会涉及：

  • crates/zeroclaw-memory/
  • 配置架构和设置文档
  • 针对 store、recall、list、forget、health、代理范围划分以及后端特定故障模式的测试。

硬件与外设

硬件集成对平台敏感，并且可能暴露真实设备。请将不安全调用、权限、设备发现和平台门控保持在硬件 crate 附近。请记录所需的操作系统软件包、udev 或权限配置步骤以及故障行为。如果某个外设通过可由 agent 调用的工具暴露，除非团队有意添加新的归属角色，否则它同样遵循 Tool 和归属约定。

何时停下来并请求 RFC

RFC 流程 是判断某项变更是否属于 RFC 范畴的权威依据。对于第一方扩展，遇到以下情况时请务必停下来对照检查：

  • 更改了多个 crate 或插件使用的公共 trait；
  • 添加影响现有用户的配置迁移；
  • 扩大文件、shell、网络、凭据或设备的访问权限；
  • 更改运行时回合执行或工具审批语义；
  • 更改了客户端使用的生成 API 契约；
  • 添加重量级依赖或新的常驻服务；
  • 创建一个新的长期所有权区域，维护者必须为其配备人员。
