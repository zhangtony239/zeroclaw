Livre de procédure de version

Processus manuel provisoire. Ce runbook explique comment publier une version stable aujourd’hui en utilisant release-stable-manual.yml. Il reste le processus en vigueur jusqu’à ce que release-plz soit déployé et le remplace ; cette migration n’a pas encore eu lieu (voir Où cela nous mène).

Si quelque chose ici vous semble lourd, il s’agit d’une friction intentionnelle : nous n’avons pas encore la discipline d’automatisation nécessaire pour la supprimer en toute sécurité.

Dernière vérification effectuée avec le cycle 0.8.0-beta.

Le processus en sept étapes

  1. Générer CHANGELOG-next.md à l’aide de la compétence changelog
  2. Ouvrir et fusionner une PR d’incrémentation de version
  3. Simuler les workflows de release localement avec act
  4. Déclencher le workflow Release Stable via un déclenchement manuel
  5. Approuver les deux barrières d’environnement lorsque vous y êtes invité
  6. Vérifier que la release existe et que les assets sont téléchargeables
  7. Déploiement de la documentation versionnée

C’est l’ensemble du processus. Tout le reste (Docker, redéploiement du site web, Scoop, AUR, Homebrew, Discord, tweet) s’exécute automatiquement sous forme de tâches en aval. Vous n’avez rien à faire pour celles-ci, sauf si une tâche échoue explicitement.

Étape 1 : Générer CHANGELOG-next.md

Exécutez le skill changelog-generation pour produire CHANGELOG-next.md. Sa procédure complète se trouve dans .claude/skills/changelog-generation/SKILL.md.

Le skill génère le changelog à partir du git log entre le dernier tag stable et HEAD, résout les contributeurs via GitHub GraphQL et écrit le fichier. Validez le résultat directement dans une branche éphémère et incluez-le dans la PR d’incrémentation de version (étape 2), ou ouvrez-le comme une PR distincte précédente si le diff est volumineux.

Si CHANGELOG-next.md existe déjà à la suite d’un cycle de publication précédemment interrompu, vérifiez son exactitude avant de le réutiliser.

Étape 2 : Incrémenter la version et fusionner la PR de version

Mettez à jour workspace.package.version dans le fichier Cargo.toml de l’espace de travail, puis exécutez les deux scripts de publication dans l’ordre. Commencez par synchroniser toutes les références de version dans l’ensemble du dépôt :

sh

./scripts/release/bump-version.sh    # version de Cargo.toml

Ceci met à jour les badges du README, la configuration Tauri et les exemples de description de workflow, puis régénère chaque surface d’installation pilotée par spec via cargo generate installers : setup.bat, dist/aur/PKGBUILD, dist/scoop/zeroclaw.json, flake.nix, les jeux de fonctionnalités Dockerfile/Containerfile, et dev/ci/docker-tags.toml. Ces surfaces dérivent leur version et leurs listes de fonctionnalités de la spec d’installation canonique (Cargo.toml plus [package.metadata.zeroclaw]), donc la mise à jour les maintient synchronisées automatiquement ; ne modifiez jamais manuellement une région générée.

Actualisez ensuite les catalogues de traduction de la documentation et épinglez-les au tag correspondant :

sh

./scripts/release/refresh-translations.sh    # initialiser le sous-module, actualiser les catalogues, étiqueter, épingler

refresh-translations.sh lit la version depuis Cargo.toml (rien n’est saisi manuellement), exécute la passe de traduction, valide et pousse les catalogues vers le sous-module zeroclaw-labs/zeroclaw-docs-translations, y crée le tag v{version}, et indexe le gitlink du dépôt principal épinglé sur ce tag. Il initialise le sous-module s’il n’est pas déjà extrait. Exécutez-le après bump-version.sh afin que la version Cargo.toml qu’il lit soit la version de release. Passez --no-translate pour ignorer la passe de synchronisation lorsque les catalogues sont déjà à jour, ou une version explicite pour remplacer la valeur par défaut de Cargo.toml.

Tout valider ensemble :

chore: bump version to vX.Y.Z

Ouvrez une PR. Étiquetez-la chore, size:XS. Obtenez la revue d’un mainteneur. Fusionnez lorsque la CI est verte. Le contrôle Installer Drift dans la CI fait échouer la PR si une surface générée est désynchronisée avec la spécification, ce qui empêche une régénération oubliée d’être intégrée. Le contrôle Validate Translations Pin résout le sous-module au commit épinglé et valide le format du catalogue ainsi que la parité des msgid, ce qui empêche également l’intégration d’un mauvais épinglage. Consultez Docs & Translations pour les détails sur l’actualisation du catalogue.

Vérifiez que la fusion s’est déroulée correctement :

sh

git fetch origin
git show origin/master:Cargo.toml | grep '^version'
# Doit afficher : version = "X.Y.Z"

Étape 3 : Effectuez une exécution à blanc des workflows de release en local avec act

Le workflow Release Stable est un graphe de jobs GitHub Actions qui consomme votre fenêtre d’approbation de la barrière d’environnement dès l’instant où vous cliquez sur Run workflow. Si une étape du workflow est défectueuse : un artefact de build manquant, un chemin obsolète, une étape de codegen que quelqu’un a supprimée sans mettre à jour la CI, l’échec apparaît après que vous vous êtes engagé dans une fenêtre de publication, avec la PR de version déjà fusionnée et master à la nouvelle version. La récupération implique de faire atterrir une branche de correctif d’urgence, de relancer la CI et de livrer sous pression temporelle sur un arbre qui se présente déjà comme une version entièrement publiée.

L’assurance à moindre coût contre cela consiste à exécuter d’abord le même graphe de jobs en local, sur le commit master fusionné exact, avant d’ouvrir le formulaire GitHub Actions. act exécute les workflows GitHub Actions dans des conteneurs Docker en utilisant le même écosystème actions/* que GitHub. Il ne reproduit pas parfaitement le runner cloud ; il ne peut pas accéder au runtime d’upload d’artefacts, aux tokens OIDC émis par GitHub, aux secrets d’environnement, ni aux jobs qui dépendent d’un vrai tag de release, mais il exécute bien les étapes de build et de test qui représentent la quasi-totalité des échecs de CI au moment des releases que nous ayons jamais rencontrés.

Cette étape représente un investissement de 15 à 20 minutes par version. Elle a permis de détecter de vrais défauts que la CI habituelle par PR ne révélait pas (car le workflow défaillant ne s’exécute que sur workflow_dispatch, pas sur push).

Configuration unique

act exécute les workflows. La méthode d’installation la plus simple est l’extension GitHub CLI, car elle hérite de votre authentification gh et expose un véritable GITHUB_TOKEN à chaque exécution de workflow :

  1. Installez la CLI GitHub depuis https://cli.github.com (Linux, macOS, Windows). Authentifiez-vous une seule fois : gh auth login.

  2. Installez l’extension act :

    sh

    gh extension install nektos/gh-act

  3. Installez Docker Engine ou Docker Desktop depuis https://docs.docker.com/engine/install/. Sous Linux, ajoutez-vous au groupe docker afin de ne pas avoir besoin de sudo. act fonctionne également avec Podman et Colima ; consultez la documentation des runners act.

C’est toute la configuration. Les fichiers .actrc et scripts/dev/act-local.sh du dépôt gèrent tout le reste (image du runner, fichier de secrets, serveur d’artefacts, pré-récupération des SHA d’action).

Simulation par version

Assurez-vous que votre arborescence de travail correspond au sommet de la branche master fusionnée de l’étape 2 :

sh

git fetch upstream
git checkout upstream/master

Liste ce qui est exécutable dans chaque fichier de workflow :

sh

./scripts/dev/act-local.sh --list

Exécuter une tâche spécifique, choisir de manière interactive ou exécuter toutes les tâches compatibles avec le mode simulation :

sh

./scripts/dev/act-local.sh release-stable-manual:web   # une tâche
./scripts/dev/act-local.sh                              # sélecteur interactif
./scripts/dev/act-local.sh --all                        # tout job sans risque en simulation (dry-run-safe)

La première exécution télécharge l’image du runner (~1,5 Go) et amorce le cache de compilation Rust via Swatinem/rust-cache ; les exécutions suivantes sont beaucoup plus rapides. Le script crée automatiquement le fichier .secrets (ignoré par git), pré-récupère chaque SHA d’action épinglé dans ~/.cache/act/ (le clone superficiel d’act ne peut pas résoudre des commits arbitraires autrement), transmet GITHUB_TOKEN depuis votre authentification gh vers l’exécution via l’environnement du processus parent (la valeur du token n’apparaît jamais dans argv), et définit --artifact-server-path pour que actions/upload-artifact et actions/download-artifact fonctionnent entre les jobs. Tout cela n’est que du act standard en dessous ; le script supprime simplement la soupe de flags.

--all exécute uniquement les tâches figurant sur une liste d’autorisation sûre en mode dry-run

act ne respecte pas les barrières de protection d’environnement de GitHub. Avec le véritable GITHUB_TOKEN du mainteneur injecté dans l’exécution, une invocation locale réussie d’un job qui écrit sur GitHub (un publish qui appelle gh release create, un job docker qui pousse vers GHCR, un docs-deploy qui force-push gh-pages, un daily-audit qui ouvre une issue, un tweet-release ou discord-release qui poste vers un webhook) pourrait produire l’effet de bord réel dès la première tentative.

--all applique donc une liste d’autorisation codée en dur de jobs dont l’exécution locale est avérée sûre ; actuellement les étapes de build produisant uniquement des artefacts dans release-stable-manual.yml et cross-platform-build-manual.yml (validate, web, release-notes, build, build-desktop). Tout le reste est ignoré avec une raison consignée dans les logs :

==> skip release-stable-manual:publish (not on dry-run-safe allowlist)
==> skip release-stable-manual:docker (not on dry-run-safe allowlist)
==> skip release-stable-manual:redeploy-website (not on dry-run-safe allowlist)
==> skip docs-deploy:deploy (not on dry-run-safe allowlist)
==> skip daily-audit:advisories (not on dry-run-safe allowlist)
==> skip tweet-release:tweet (not on dry-run-safe allowlist)

La liste d’autorisation est fail-closed (fermée par défaut) : un nouveau workflow ajouté au dépôt est traité comme potentiellement mutateur jusqu’à ce qu’un mainteneur l’examine et ajoute les identifiants de jobs sûrs à DRY_RUN_SAFE_JOBS dans scripts/dev/act-local.sh. C’est important car discover_jobs parcourt tous les fichiers .github/workflows/*.yml, et pas seulement les workflows de release, une liste de blocage laisserait silencieusement passer un futur workflow avec surface d’écriture.

Deux solutions de contournement existent pour le cas rare où vous avez une raison de tenter d’exécuter localement un job qui ne figure pas dans la liste d’autorisation :

  • ./scripts/dev/act-local.sh release-stable-manual:publish : la forme explicite <wf>:<job> exécute ce que vous demandez et affiche un avertissement bien visible avant d’invoquer act si la cible ne figure pas dans la liste d’autorisation.
  • ./scripts/dev/act-local.sh --all --no-allowlist : désactive le filtre de liste d’autorisation pour l’ensemble d’une exécution --all (à utiliser uniquement lorsque vous avez déjà vérifié que les étapes du workflow n’atteindront pas une surface de mutation, par exemple sur un fork sans véritables identifiants de registre et avec un fichier .secrets vide).

Ce qui est censé échouer sous act (et c’est normal)

act ne peut pas simuler quelques fonctionnalités spécifiques à GitHub. Ces échecs ne sont pas de véritables défauts :

  • Jobs qui dépendent d’un véritable tag de release (publish créant une GitHub Release).
  • Jobs conditionnés par un environnement (publish, docker) : l’interface d’approbation n’existe pas en local.
  • Jetons d’identité fédérée basés sur OIDC.

Tout le reste, une erreur tsc, un fichier manquant, un échec de compilation Rust, une incohérence du fichier de verrouillage cargo, est un défaut réel. Ne cliquez pas sur Run workflow dans le formulaire GitHub Actions tant que ces problèmes ne sont pas corrigés via une PR standard à partir de master.

Étape 4 : Déclencher la publication

Aller à :

https://github.com/zeroclaw-labs/zeroclaw/actions/workflows/release-stable-manual.ymlCliquez sur Run workflow. Renseignez :

  • Branch : master
  • Version stable à publier : X.Y.Z, sans préfixe v

Cliquez sur Run workflow.

Le premier job (validate) vérifie que la version correspond à Cargo.toml et qu’aucun tag vX.Y.Z n’existe déjà. En cas d’échec, corrigez l’incohérence et relancez. N’essayez pas de le contourner.

Étape 5 : Approuver les portes d’environnement

Deux jobs sont conditionnés par des règles de protection d’environnement GitHub. Lorsque chacun passe en attente, vous verrez une bannière “Waiting for review” dans l’exécution du workflow.

Approuvez les deux lorsqu’ils apparaissent :

EnvironnementTâcheCe que cela fait
github-releasespublishCrée la GitHub Release et téléverse les assets
dockerdockerPousse les images vers GHCR

Si vous manquez la fenêtre d’approbation et qu’une tâche expire, relancez uniquement la tâche échouée depuis la page d’exécution du workflow ; vous n’avez pas besoin de tout recommencer depuis le début.

Étape 6 : Vérifier la version publiée

Une fois publish terminé, confirmez :

[ ] GitHub Release exists at /releases/tag/vX.Y.Z and is marked Latest
[ ] Release notes are non-empty
[ ] SHA256SUMS asset is present and non-empty
[ ] At least one binary archive is downloadable (spot-check linux x86_64)

CHANGELOG-next.md est intentionnellement conservé sur master après la publication : le job de publication se contente de le lire comme corps de la release, il ne le supprime pas. Le cycle de publication suivant l’écrase, donc aucun nettoyage manuel n’est nécessaire.

Vous n’avez pas besoin de vérifier manuellement Docker ou les canaux de distribution, sauf si un job de l’exécution du workflow apparaît en rouge. Consultez le résumé de l’exécution du workflow ; si tous les jobs sont au vert, vous avez terminé.

Étape 7 : Déploiement de la documentation versionnée

La documentation de ZeroClaw utilise une structure versionnée sur la branche gh-pages. Lorsqu’un tag est poussé, le workflow Deploy mdBook docs to Pages construit et déploie automatiquement la documentation pour cette version. Cela s’exécute automatiquement après l’arrivée du tag de l’étape 4 ; les détails ci-dessous sur le bootstrap et la version minimale sont des informations de référence pour les cas où vous devez recréer gh-pages ou modifier la fenêtre des versions prises en charge.

Ce qui se passe automatiquement

  • Le push d’un tag (par exemple, v0.8.0) déclenche un build qui est publié dans /v0.8.0/.
  • « Stable » est un pointeur, pas une copie. C’est le déploiement du tag de release (par ex. v0.8.0) qui construit et publie le répertoire de docs de cette version. bump-version.sh écrit la version publiée dans docs/book/stable-version.txt ; intégrer ce changement sur master rafraîchit uniquement les métadonnées stable. Le déploiement de master ne reconstruit ni ne republie les docs du tag de release ; il copie stable-version.txt à la racine gh-pages et régénère la redirection racine / ainsi que l’entrée « Stable (latest release) » du sélecteur de version, afin que les deux pointent vers le répertoire de version déjà publié de cette release. Le déploiement échoue de manière explicite si le répertoire de version nommé n’est pas présent sur gh-pages. Il n’y a pas d’arborescence /stable/ en double.
  • gh-pages est éphémère : chaque déploiement effectue un force-push d’un unique commit orphelin (sans historique cumulé) et applique la rétention via DOCS_KEEP_VERSIONS (master plus les N versions finales les plus récentes ; les pré-versions et les anciennes versions finales sont élaguées). Cela permet de limiter la taille du clone.
  • Le répertoire _shared/ (contenant les CSS de l’interface, les JS et les favicons) est mis à jour à partir du build afin que le thème se propage à toutes les versions déployées.
  • Les langues traduites (es, fr, ja, zh-CN) sont générées à partir du sous-module docs/book/po, que le déploiement résout via submodules: recursive au commit épinglé par la référence déployée. Cet épinglage est défini lors de l’incrémentation de version ; voir Step 2 pour la procédure de rafraîchissement, de marquage et d’épinglage. L’anglais ne nécessite aucun sous-module.

Amorçage de gh-pages

Si gh-pages est un jour supprimée ou doit être entièrement recréée, initialisez les versions dans cet ordre précis :

  1. Version la plus ancienne prise en charge : workflow_dispatch avec le tag v0.7.5
  2. Prochaines versions : workflow_dispatch avec le tag v0.8.0-beta-1, etc.
  3. Master actuel : workflow_dispatch avec le tag master

[!IMPORTANT] master doit être déployé en dernier lors du bootstrapping. Il écrit la couche de chrome _shared/ définitive utilisée par toutes les autres versions.

[!NOTE] La version stable est résolue depuis docs/book/stable-version.txt (validé dans le code source, publié à la racine gh-pages sous stable-version.txt). Après l’amorçage, vérifiez que ce fichier référence bien la version GA prévue ; la redirection à la racine et l’entrée du sélecteur « Stable (latest release) » s’appuient dessus. Aucun répertoire /stable/ n’est créé.

Redéploiements manuels et le plancher de version

Pour redéployer manuellement une version spécifique :

  1. Allez dans Actions → Deploy mdBook docs to Pages
  2. Cliquez sur Run workflow
  3. Entrez le tag (par exemple, v0.7.5 ou master)

Le plancher DOCS_MIN_VERSION : Pour éviter de déployer accidentellement des versions très anciennes ou non prises en charge, le workflow applique un plancher de version minimale (actuellement v0.7.5).

  • Les tags plus anciens que DOCS_MIN_VERSION (comme v0.7.4) sont rejetés par le workflow.
  • cargo mdbook gen-versions (l’assistant xtask) ignore tous les répertoires sur gh-pages situés en dessous de ce seuil, les excluant ainsi du menu déroulant des versions.

Si vous devez relever le seuil minimal pour abandonner la prise en charge d’une version plus ancienne :

  1. Mettez à jour la variable d’environnement DOCS_MIN_VERSION dans .github/workflows/docs-deploy.yml.
  2. Les répertoires des anciennes versions sont automatiquement supprimés lors du prochain déploiement par la passe de rétention DOCS_KEEP_VERSIONS ; aucune modification manuelle de gh-pages n’est requise pour libérer de l’espace.

Si quelque chose ne va pas

échec de la validation : incompatibilité de version : La PR de montée de version n’a pas été fusionnée, ou vous avez saisi une version incorrecte. Corrigez l’incompatibilité et relancez le déclenchement.

Délai d’attente dépassé pour une barrière d’environnement : Relancez uniquement la tâche en échec. Inutile de redémarrer le workflow.

Échec d’un job de canal de distribution (Scoop, AUR, Homebrew) : Chacun dispose d’un sous-workflow correspondant pouvant être déclenché manuellement. Relancez celui concerné avec dry_run: true d’abord pour confirmer le correctif, puis avec dry_run: false. Ces jobs sont optionnels : un échec du job Scoop n’invalide pas la release elle-même.

Suppression des workflows hérités

Plusieurs workflows de publication automatique qui se trouvaient auparavant dans .github/workflows/ ont été supprimés car ils contournaient la revue ou publiaient de manière irréversible. Ils ne sont plus présents ; si l’un d’eux réapparaît dans une PR, traitez-le comme une régression et bloquez-la :

Flux de travailPourquoi cela a été supprimé
release-beta-on-push.ymlPublié automatiquement à chaque push sur master
publish-crates-auto.ymlPublié automatiquement sur crates.io à chaque changement de version, irréversible
version-sync.ymlCommit effectué directement sur master en tant que bot, en contournant la revue
checks-on-pr.ymlCI en double : produisait un statut contradictoire et déroutant
pre-release-validate.ymlListe de contrôle générée inutilisée ; ce runbook la remplace

L’inventaire complet des workflows restants (automatiques et manuels) se trouve dans CI & Actions.

Vers quoi cela évolue

Ce runbook et release-stable-manual.yml sont une transition, pas une finalité.

L’état final cible :

  • release-plz gère automatiquement les incréments de version et les changelogs
  • Un seul fichier release.yml remplace l’assemblage hétéroclite actuel de sous-workflows
  • La provenance SLSA est intégrée au pipeline
  • L’équipe publie les versions en fusionnant une PR de release, et non en suivant un runbook

En attendant que cela soit en place, utilisez ce processus. Chaque version que vous publiez manuellement à l’aide de ce runbook constitue un exercice qui éclaire ce que l’automatisation devra accomplir.
