ZeroClaw Docs

Docker & Conteneurs

Exécutez ZeroClaw dans Docker, Podman, Kubernetes ou n’importe quel runtime OCI.

Images officielles

Publié sur GitHub Container Registry (ghcr.io) à chaque version stable :

  • ghcr.io/zeroclaw-labs/zeroclaw:latest : dernière version stable
  • ghcr.io/zeroclaw-labs/zeroclaw:v0.7.5 : épinglée
  • ghcr.io/zeroclaw-labs/zeroclaw:debian : image basée sur Debian (plus volumineuse, prise en charge plus large de glibc)

Multi-arch : linux/amd64, linux/arm64.

Remarque sur l’accès au shell : L’image latest par défaut est intentionnellement distroless et n’inclut pas sh, ash ni bash. Utilisez le tag debian si vous avez besoin d’un shell à l’intérieur du conteneur (par exemple, pour exécuter docker exec à des fins de débogage).

Exécution minimale

sh

docker run -d \
  --name zeroclaw \
  -v zeroclaw-data:/zeroclaw-data \
  -p 42617:42617 \
  ghcr.io/zeroclaw-labs/zeroclaw:latest

L’image officielle lie déjà [::] avec allow_public_bind = true et require_pairing = false intégrés dans sa configuration par défaut, donc le port publié est accessible dès le départ. La surcharge ZEROCLAW_gateway__allow_public_bind ci-dessous n’a d’importance que si vous montez en bind votre propre configuration (qui remplace celle intégrée) dont la valeur par défaut est localhost.

L’image s’attend à un état persistant dans /zeroclaw-data. Au premier lancement, elle initialise une configuration par défaut : vous devez tout de même exécuter quickstart avant qu’elle ne soit utilisable :

sh

docker exec -it zeroclaw zeroclaw quickstart

Exécution de zerocode (la TUI)

L’image embarque l’interface de terminal zerocode aux côtés du binaire zeroclaw. Le point d’entrée par défaut est zeroclaw, donc lancez zerocode en le remplaçant avec --entrypoint zerocode et un TTY interactif (-it). Les deux variantes de l’image l’incluent :

distroless (:latest)

docker run -it --entrypoint zerocode ghcr.io/zeroclaw-labs/zeroclaw:latest

debian

docker run -it --entrypoint zerocode ghcr.io/zeroclaw-labs/zeroclaw:debian

zerocode se connecte à un démon ZeroClaw en cours d’exécution, il faut donc le faire pointer vers l’un d’eux :

  • Daemon du même conteneur : exécutez-le contre le conteneur qui exécute déjà le daemon (docker exec -it zeroclaw zerocode), qui atteint le daemon via le socket IPC local.
  • Un démon distant : connectez-vous via WebSocket Secure avec zerocode --connect wss://<host>:<port> ; voir Configuration à distance (WSS). C’est la méthode portable pour piloter un démon conteneurisé ou distant depuis votre propre terminal.

Persistez /zeroclaw-data (comme dans Minimum run) afin que la configuration et l’identité lues par zerocode soient les mêmes que celles utilisées par le démon.

Composer

Un docker-compose.yml minimal :

services:
  zeroclaw:
    image: ghcr.io/zeroclaw-labs/zeroclaw:latest
    redémarrer: sauf-arrêt
    ports:
      - "42617:42617"      # passerelle
    volumes:
      - ./data:/zeroclaw-data
    # L'image officielle active déjà la liaison publique ; ajoutez uniquement un bloc `environment:`
    # avec la valeur de remplacement ci-dessus si vous montez en bind une configuration par défaut localhost.

Après le démarrage du conteneur, exécutez quickstart :

sh

docker compose exec zeroclaw zeroclaw quickstart

Avec l’image officielle, vous pouvez omettre entièrement la surcharge ZEROCLAW_gateway__allow_public_bind ; elle est déjà activée dans la configuration intégrée.

Compose sans root avec l’image Debian

Pour les déploiements rootless avec Docker ou Podman Compose qui nécessitent des outils shell à l’intérieur du conteneur, utilisez l’image Debian actuelle et liez un répertoire de données de l’hôte :

services:
  zeroclaw:
    image: ghcr.io/zeroclaw-labs/zeroclaw:debian
    container_name: zeroclaw
    redémarrer: sauf-arrêt
    ports:
      - "42617:42617"
    volumes:
      - ./data:/zeroclaw-data
    vérification de l'état:
      test: [CMD, zeroclaw, "statut", --format=exit-code]
      interval: 60s
      timeout: 10s
      nouvelles tentatives: 3
      start_period: 10s

L’image Debian actuelle place le tableau de bord packagé en dehors de /zeroclaw-data, de sorte que le bind mount ne le masque pas et qu’aucune surcharge gateway.web_dist_dir n’est nécessaire. L’image officielle embarque également les valeurs par défaut de bind de la gateway adaptées aux conteneurs. N’ajoutez un bloc environment: que si vous bind-montez votre propre configuration par défaut sur localhost ; utilisez l’orthographe des variables d’environnement reflétant le schéma, comme illustré par ZEROCLAW_gateway__allow_public_bind, plutôt que les anciens alias entièrement en majuscules.

macOS : OrbStack vs Colima

macOS ne possède pas de noyau Linux natif, donc chaque option (Docker Desktop, Podman, OrbStack, Colima) exécute le conteneur dans une VM Linux légère. Pour une machine de développement Mac, les deux VM natives Mac qui méritent d’être comparées sont OrbStack et Colima, toutes deux exécutent le conteneur avec les mêmes commandes docker run/Compose ci-dessus.

OrbStackColima
MoteurVM Linux personnalisée et optimisée (optimisée pour Apple Silicon)Lima VM + containerd/Docker
Licencecommercial, freemium (utilisation personnelle gratuite)MIT (Lima sous-jacent est sous licence Apache 2.0)
InterfaceApplication graphique + CLIPrivilégie la CLI (colima start/stop), scriptable
Idéal lorsquesans complication, expérience utilisateur soignéetout en OSS, configuration dans le code

OrbStack

# Fournit la CLI docker :
brew install --cask orbstack

Colima

# La CLI docker communique avec la VM de colima :
brew install colima docker docker-compose   # docker-compose = le plugin Compose v2 ; à installer si vous avez besoin de `docker compose`
colima start --cpu 4 --memory 8   # ajouter --network-address pour exposer l'IP de la VM à macOS

Les performances sont comparables pour les charges de travail de développement typiques ; les véritables différenciateurs sont la licence (commerciale vs OSS) et les préférences UX, pas la vitesse brute ; effectuez un benchmark des deux sur votre propre machine si la RAM au repos ou le débit de build est important pour vous. Dans tous les cas, vous pilotez le moteur à l’intérieur de la VM avec docker ; les quadlets systemd (ci-dessous) sont une fonctionnalité des hôtes Linux et ne s’appliquent pas sur macOS.

Podman & quadlets systemd

Sur un serveur Linux, la façon la plus propre d’exécuter le conteneur sur le long terme est un quadlet Podman : un fichier d’unité déclaratif que systemd transforme en un véritable service. Vous bénéficiez du cycle de vie systemctl, des journaux journald, du redémarrage automatique et de l’ordonnancement au démarrage, sans daemon et sans bidouille --restart, et le fichier d’unité est une configuration que vous committez dans git. C’est le modèle recommandé pour les serveurs ; docker run/Compose conviennent très bien pour un ordinateur portable.

Un quadlet est un fichier *.container (fichiers apparentés : .pod, .volume, .network, .kube, .build, .image). Le générateur systemd de Podman le lit à chaque daemon-reload et écrit un fichier .service transitoire ; vous ne rédigez jamais le .service vous-même.

Les unités rootful résident dans /etc/containers/systemd/ ; les unités rootless dans ~/.config/containers/systemd/.

/etc/containers/systemd/zeroclaw.container :

[Unit]
Description=ZeroClaw agent runtime
After=network-online.target
Wants=network-online.target

[Container]
# Pin a release in production; :latest is distroless (no shell — use :debian to exec a shell).
Image=ghcr.io/zeroclaw-labs/zeroclaw:latest
ContainerName=zeroclaw
PublishPort=42617:42617
Volume=zeroclaw-data:/zeroclaw-data
# The official image already binds publicly; add an `Environment=` line with the
# allow-public-bind override only if you mount a localhost-default config.
# Optional rolling-upgrade path — re-pull a newer image on (re)start and opt into `podman auto-update`:
Pull=newer
AutoUpdate=registry

[Service]
Restart=always

[Install]
WantedBy=multi-user.target default.target

Déployer (idempotent, réexécutable sans risque ; une nouvelle application fait converger le conteneur en cours d’exécution, sans jamais le dupliquer) :

sh

sudo cp zeroclaw.container /etc/containers/systemd/
sudo systemctl daemon-reload      # le générateur transforme .container en zeroclaw.service
sudo systemctl restart zeroclaw

Ensuite, effectuez l’intégration une seule fois, puis gérez-le comme n’importe quel service :

sh

sudo podman exec -it zeroclaw zeroclaw quickstart
systemctl status zeroclaw
journalctl -u zeroclaw -f

Il n’y a pas d’étape systemctl enable pour les unités générées : c’est la ligne [Install] WantedBy= qui assure son démarrage au boot.

  • Épinglage de version vs :latest. Épinglez un tag ou un digest (Image=ghcr.io/zeroclaw-labs/zeroclaw:v0.7.5 ou ...@sha256:...) pour des déploiements reproductibles et auditables ; la mise à niveau se résume alors à un changement de tag révisable dans le fichier .container commité. Pull=newer + AutoUpdate=registry offrent à la place des mises à niveau continues, pilotées par podman-auto-update.timer (sudo systemctl enable --now podman-auto-update.timer). Choisissez entre reproductibilité et actualité ; la boucle de déploiement reste la même dans les deux cas.
  • Variante rootless. Déposez le fichier dans ~/.config/containers/systemd/, utilisez systemctl --user daemon-reload && systemctl --user restart zeroclaw, et exécutez loginctl enable-linger $USER pour qu’il survive à la déconnexion (même remarque sur le lingering que dans Service & daemon).
  • WSL2. Les versions modernes de WSL2 exécutent systemd ([boot] systemd=true dans /etc/wsl.conf, puis wsl --shutdown), donc ce modèle quadlet exact fonctionne dans une distribution WSL : aucun dialecte spécifique à Windows.

Configuration à l’intérieur des conteneurs

L’image s’attend à trouver la configuration sous /zeroclaw-data/.zeroclaw/. Montez votre configuration locale dans :

sh

docker run -d --name zeroclaw \
  -v $(pwd)/my-config.toml:/zeroclaw-data/.zeroclaw/config.toml:ro \
  -v zeroclaw-state:/zeroclaw-data/workspace \
  -p 42617:42617 \
  ghcr.io/zeroclaw-labs/zeroclaw:latest

Pour les charges de travail conteneurisées, définissez uri sur chaque providers.models.<type>.<alias> avec une adresse accessible depuis le conteneur (p. ex. http://host.docker.internal:11434 pour un serveur Ollama sur l’hôte Docker Desktop). Le mécanisme générique de surcharge par variables d’environnement peut définir le même champ à l’exécution sans modifier la configuration :

sh

ZEROCLAW_providers__models__ollama__home__uri=http://ollama:11434 zeroclaw agent -a assistant

Voir Fournisseurs → Surcharges adaptées aux conteneurs pour la grammaire.

Les canaux fonctionnant par sondage (Telegram, e-mail) : fonctionnent directement

Les canaux initiés en sortie ne nécessitent aucune configuration particulière du conteneur. Polling Telegram, IMAP, MQTT, relais Nostr : tout fonctionne en mode pull ; le conteneur n’a besoin que d’un accès sortant (egress).

Canaux qui reçoivent des webhooks : nécessitent un ingress

Discord, Slack, GitHub et la plupart des canaux webhook nécessitent un accès HTTP entrant. Deux options :

  1. Exposer la passerelle : -p 42617:42617 + un proxy inverse avec TLS en frontal, faire pointer l’URL du webhook vers l’adresse publique
  2. Utiliser un tunnel : ngrok, Cloudflare Tunnel ou Tailscale Funnel ; définissez l’URL du tunnel comme cible du webhook

Configurez un tunnel en définissant le tunnel_provider de niveau supérieur [tunnel] (variable d’environnement de remplacement : ZEROCLAW_tunnel__tunnel_provider) sur l’un des fournisseurs pris en charge et en remplissant le bloc tunnel.* correspondant ; la liste complète des fournisseurs et les champs propres à chaque fournisseur se trouvent dans la Référence de configuration. L’URL publique obtenue est celle vers laquelle vous devez diriger vos émetteurs de webhooks.

Kubernetes

Les modèles de chart Helm sont publiés dans le dépôt zeroclaw-templates. Fragment de manifeste typique :

apiVersion: apps/v1
kind: Déploiement
métadonnées:
  nom: zeroclaw
spécification:
  répliques: 1
  stratégie:
    type: Recreate         # ZeroClaw est une instance unique par espace de travail
  template:
    spécification:
      conteneurs:
        - nom: zeroclaw
          image: ghcr.io/zeroclaw-labs/zeroclaw:v0.7.5
          ports:
            - `containerPort`: 42617
          volumeMounts:
            - nom: données
              chemin de montage: /zeroclaw-data
          # L'image officielle effectue déjà une liaison publique ; ajoutez une entrée `env:` pour la
          # dérogation allow-public-bind uniquement si vous montez une configuration localhost par défaut.
      volumes:
        - nom: données
          persistentVolumeClaim:
            nom de la revendication: zeroclaw-data

Mise à l’échelle : ZeroClaw fonctionne avec un seul rédacteur par espace de travail. Ne procédez pas à une mise à l’échelle horizontale ; exécutez une instance par agent.

Réauthentification après déconnexion

Si vous vous déconnectez de l’interface web pendant l’exécution dans un conteneur, le paircode existant devient invalide. Générez-en un nouveau pour vous reconnecter :

sh

docker exec -it zeroclaw zeroclaw gateway get-paircode --new

Pour les déploiements Compose, utilisez plutôt docker compose exec :

sh

docker compose exec zeroclaw zeroclaw gateway get-paircode --new

Pièges

  • Particularités de nom d’hôte sur macOS (Docker Desktop, colima, Rancher Desktop). host.docker.internal fonctionne d’emblée sur Docker Desktop pour macOS. Sur colima, il n’est accessible que si vous avez effectué l’installation avec colima start --network-address (sinon le conteneur ne peut pas du tout voir l’hôte ; connectez-vous via l’IP de passerelle de la VM, généralement 192.168.5.2, ou créez un tunnel via un réseau partagé). Rancher Desktop se comporte comme Docker Desktop dans les versions récentes, mais a connu des échecs de résolution de host.docker.internal dans des versions plus anciennes. Si les appels au fournisseur échouent avec connection refused vers host.docker.internal, vérifiez avec docker run --rm alpine getent hosts host.docker.internal : une sortie vide signifie que le nom d’hôte n’est pas résolvable et que vous devez utiliser une IP explicite.
  • Services côté hôte. Si un fournisseur est Ollama sur l’hôte, uri = "http://host.docker.internal:11434" (sous [providers.models.ollama.<alias>]) fonctionne sur Docker Desktop. Sur Linux Docker, vous pourriez avoir besoin de --add-host=host.docker.internal:host-gateway.
  • Persistance de la mémoire. La mémoire de l’agent (la base SQLite brain.db) se trouve dans le répertoire de configuration à l’emplacement /zeroclaw-data/.zeroclaw/agents/<alias>/workspace/memory/, avec les bases de données d’instance partagées sous /zeroclaw-data/data/. Monter /zeroclaw-data permet de tout persister ; sans ce volume, chaque redémarrage entraîne la perte de l’historique des conversations.
  • Montage bind sur /zeroclaw-data. Un montage bind de l’hôte sur /zeroclaw-data remplace l’intégralité du répertoire de l’image, y compris la configuration par défaut et (auparavant) le bundle du tableau de bord. Le tableau de bord est désormais installé dans /usr/share/zeroclawlabs/web/dist, en dehors du montage, de sorte qu’un montage bind ne le masque plus. Au premier lancement, montez un répertoire hôte vide et le conteneur initialise une configuration neuve ; la passerelle détecte automatiquement le tableau de bord depuis son chemin dans l’image.
  • Aucun transfert matériel par défaut. GPIO / USB nécessitent des indicateurs --device explicites (--device /dev/ttyUSB0), et l’utilisateur du conteneur doit avoir le GID correspondant pour les groupes dialout/gpio.
