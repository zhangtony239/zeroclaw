ZeroClaw Docs

Windows

Installer, mettre à jour, exécuter en tant que tâche planifiée Windows et désinstaller sur Windows 10 / 11.

Si vous utilisez WSL2, vous pouvez suivre la configuration Linux à la place ; install.sh s’exécute sans modification sous WSL.

Note sur setup.bat. Les échecs bloquants de #6118 (le dépassement d’espace disque dû à set /a en 32 bits et l’erreur d’analyse if/else due aux parenthèses non échappées) ont été corrigés dans #6137 et sont disponibles à partir de la v0.7.4. Les versions actuelles se terminent normalement. Une réserve subsiste : setup.bat --prebuilt vérifie toujours la présence de cargo avant d’atteindre la branche prebuilt, de sorte que le chemin prebuilt manuel (Option 1 ci-dessous) constitue la véritable installation sans Rust. La compilation depuis les sources (Option 3) fonctionne également.

Installer

Option 1 : Binaire précompilé (recommandé)

Téléchargez le fichier zip de la dernière version Windows, extrayez zeroclaw.exe, et placez-le dans votre PATH.

À partir d’une invite PowerShell :

# Idempotent : réexécuter ce bloc est sans effet lorsque zeroclaw est déjà
# installé dans sa dernière version et présent dans le PATH de l'utilisateur. Après
# une montée de version, la vérification de version échoue et l'installation se relance.
$ver = (Invoke-RestMethod 'https://api.github.com/repos/zeroclaw-labs/zeroclaw/releases/latest').tag_name.TrimStart('v')
$dst = "$env:USERPROFILE\.zeroclaw\bin"
$exe = "$dst\zeroclaw.exe"

$current = if (Test-Path $exe) {
    ((& $exe --version 2>$null) | Select-String -Pattern '\d+\.\d+\.\d+').Matches.Value
} else { '' }

if ($current -ne $ver) {
    $url = "https://github.com/zeroclaw-labs/zeroclaw/releases/download/v$ver/zeroclaw-x86_64-pc-windows-msvc.zip"
    New-Item -ItemType Directory -Force -Path $dst | Out-Null
    Invoke-WebRequest -Uri $url -OutFile "$env:TEMP\zeroclaw.zip" -UseBasicParsing
    Expand-Archive -Force -Path "$env:TEMP\zeroclaw.zip" -DestinationPath $dst
}

$userPath = [Environment]::GetEnvironmentVariable('Path', 'User')
if (($userPath -split ';') -notcontains $dst) {
    [Environment]::SetEnvironmentVariable('Path', "$dst;$userPath", 'User')
}
if (($env:Path -split ';') -notcontains $dst) {
    $env:Path = "$dst;$env:Path"
}

& $exe quickstart

Le fichier zip fournit un binaire autonome, sans toolchain Rust ni Visual Studio Build Tools nécessaires.

Après l’installation, vérifiez :

zeroclaw --version    # matches the latest release

Option 2 : setup.bat (à partir d’une version)

setup.bat --prebuilt

Indicateurs :

IndicateurComportement
--prebuiltTélécharger le binaire précompilé depuis GitHub Releases (le plus rapide une fois disponible ; le script actuel vérifie toujours cargo en premier)
--minimalCompiler uniquement le cœur (sans canaux, sans matériel)
--standardCréez avec des canaux courants (Telegram, Discord, Slack, Matrix)
--fullTout générer

⚠️ Problème connu (actuel). setup.bat --prebuilt vérifie toujours la présence de cargo avant d’atteindre la branche prebuilt, donc l’option 2 ne respecte pas la promesse d’absence de Rust. Si vous n’avez pas de chaîne d’outils Rust, utilisez l’option 1 ci-dessus.

Historique (avant v0.7.4). Les versions antérieures présentaient deux échecs bloquants et une incohérence de commande d’intégration signalés dans #6118 : un dépassement de capacité d’espace disque set /a sur 32 bits (Invalid number. Numbers are limited to 32-bits of precision.), une erreur d’analyse if/else due à des parenthèses non échappées (.[0m was unexpected at this time.) et une invite finale zeroclaw init. Tous ont été corrigés dans #6137 (v0.7.4 / 0.7.5 / 0.8.0) ; les versions actuelles affichent zeroclaw quickstart et se terminent normalement.

Option 3 : À partir des sources

Requiert Rust (rustup) et les Outils de génération Visual Studio :

git clone https://github.com/zeroclaw-labs/zeroclaw
cd zeroclaw
cargo install --locked --path .
zeroclaw quickstart

Option 4 : Scoop (actuellement obsolète)

⚠️ Le manifeste Scoop dans le dépôt est épinglé à la v0.5.9 (23 versions correctives de retard sur master). Tant qu’un hook CI au moment de la publication ne le met pas à jour, préférez l’Option 1 ou 3. Si vous utilisez Scoop et rencontrez des problèmes, veuillez ouvrir une PR sur dist/scoop/zeroclaw.json.

scoop install zeroclaw     # currently installs an older release; see warning above
zeroclaw quickstart

Option 5 : Docker

ZeroClaw publie une image de conteneur Linux sur ghcr.io/zeroclaw-labs/zeroclaw:latest (et :vX.Y.Z pour les versions taguées). Sous Windows, exécutez-la via Docker Desktop ou via sudo apt install docker.io dans une distribution WSL ; les deux fonctionnent, et le comportement du conteneur est identique.

Démarrage rapide :

# Volume persistant pour la configuration + l'espace de travail ; le répertoire de données de ZeroClaw à l'intérieur du conteneur est /zeroclaw-data
docker run -d --name zeroclaw `
  --restart=unless-stopped `
  -p 42617:42617 `
  -v zeroclaw-data:/zeroclaw-data `
  ghcr.io/zeroclaw-labs/zeroclaw:latest

# Suivre les journaux de premier démarrage
docker logs -f zeroclaw

# Vérification de l'état (aucune authentification requise)
curl http://localhost:42617/health

# (Facultatif) appairer un client. REMARQUE : l'image publiée définit par défaut
# `require_pairing = false`, donc par défaut aucun code d'appairage n'est émis
# et `/api/*` accepte les requêtes sans authentification. Pour activer l'appairage,
# remplacez la configuration (définissez `require_pairing = true` dans
# /zeroclaw-data/.zeroclaw/config.toml à l'intérieur du conteneur) et redémarrez ;
# un code à usage unique est alors affiché sur stdout au premier démarrage, après quoi
# les clients l'envoient via POST à `/pair` :
curl -X POST http://localhost:42617/pair -H 'X-Pairing-Code: <code-from-logs>'

Faits relatifs à l’image (vérifiés par rapport à ghcr.io/zeroclaw-labs/zeroclaw:latest) :

  • Base : gcr.io/distroless/cc-debian13:nonroot (étape de release ; l’étape dev est debian:trixie-slim)
  • ENTRYPOINT ["zeroclaw"], CMD ["daemon"] : l’exécution sans argument démarre le daemon et la passerelle
  • EXPOSE 42617 : le daemon et la passerelle écoutent tous deux sur ce port
  • Répertoire de données : /zeroclaw-data (config : /zeroclaw-data/.zeroclaw/config.toml, espace de travail : /zeroclaw-data/workspace). Montez un volume nommé ou un bind ici pour la persistance ; notez qu’il ne s’agit pas de /root/.zeroclaw.
  • Appairage : l’image publiée utilise par défaut require_pairing = false, donc /api/* accepte les requêtes sans authentification d’emblée. Lorsque l’appairage est activé (définissez require_pairing = true dans /zeroclaw-data/.zeroclaw/config.toml et redémarrez), le démon affiche un code à usage unique sur stdout au premier démarrage, et les clients l’envoient ensuite par POST à /pair avec l’en-tête X-Pairing-Code avant que tout point de terminaison authentifié ne réponde.
  • Tableau de bord web : intégré et servi par défaut dans l’image publiée. L’image définit gateway.web_dist_dir = "/usr/share/zeroclawlabs/web/dist" et y inclut le frontend compilé, de sorte que le gateway sert le fallback SPA clé en main. Les assets se trouvent en dehors du point de montage /zeroclaw-data, de sorte qu’un montage de volume -v …:/zeroclaw-data ne peut pas les masquer (réf. #6400).

Compilez à partir du source en utilisant le Dockerfile fourni :

git clone https://github.com/zeroclaw-labs/zeroclaw
cd zeroclaw
docker build -t zeroclaw:local -f Dockerfile.debian .

Vérifié sous Windows + Docker :

  • Le comportement du conteneur correspond à celui de Linux. Image ghcr.io/zeroclaw-labs/zeroclaw:latest téléchargée et exécutée sous WSL Debian sur Windows 11 build 26200.8313. L’image démarre proprement, la passerelle écoute sur :42617, et /health renvoie un JSON valide. Avec require_pairing = true défini dans la configuration et le conteneur redémarré, le flux de code d’appairage sur /pair fonctionne également comme documenté.
  • Docker sans Docker Desktop. wsl --install pour activer WSL2, puis sudo apt install docker.io dans la distribution WSL, vous donne le démon directement ; vérifié pour récupérer et exécuter l’image publiée sans modification.

Bonnes pratiques côté hôte : recommandations générales pour Docker + WSL2, et non des affirmations spécifiques au runtime de zeroclaw. Sourcées depuis Microsoft Learn et la documentation officielle de Docker le cas échéant :

  • Montages de volumes. Le bind-mount de chemins côté Windows (-v C:/Users/...:/zeroclaw-data) dans un conteneur Linux franchit la limite du système de fichiers WSL2 ⇄ Windows ; Microsoft documente la disposition et les implications des chemins inter-OS dans la référence WSL file systems. Préférez les volumes nommés Docker (-v zeroclaw-data:/zeroclaw-data) ou stockez l’espace de travail dans le système de fichiers WSL (\\wsl$\Debian\home\...) pour des performances quasi natives.

  • Réseau. Par défaut, le réseau WSL2 utilise du NAT, donc les services dans le conteneur sont accessibles depuis Windows via localhost:<port> après redirection avec -p (vérifié sur Windows 11 + WSL2). Si vous devez accéder au conteneur depuis une autre machine du réseau local, ou exécuter des configurations multi-conteneurs où le DNS inter-conteneurs est important, passez en mode miroir selon la référence Mirrored mode networking de Microsoft, en ajoutant à %USERPROFILE%\.wslconfig :

    [wsl2]
networkingMode=mirrored

  • Démon sous Docker, pas Planificateur de tâches. À l’intérieur du conteneur, il n’y a pas de Planificateur de tâches Windows. Utilisez la politique de redémarrage de Docker, --restart=unless-stopped comme dans l’exemple ci-dessus, pour le démarrage en mode démon. L’image publiée s’exécute en tant que PID 1 / utilisateur nonroot ; le conteneur est le service ; n’exécutez pas zeroclaw service install à l’intérieur.

  • Sandbox de compétences via le socket Docker hôte. Le sandbox d’exécution de compétences de ZeroClaw peut faire appel à Docker. Si vous exécutez ZeroClaw lui-même dans un conteneur et que vous souhaitez que le sandbox de compétences utilise également Docker, montez le socket Docker hôte afin que les conteneurs enfants s’exécutent sur le daemon hôte plutôt que d’imbriquer Docker-in-Docker :

    # PowerShell / cmd.exe : utilisez une seule barre oblique au début.
# Git Bash / MINGW : utilisez //var/run/docker.sock pour contourner la réécriture de chemin MSYS.
-v /var/run/docker.sock:/var/run/docker.sock

    Sachez que monter le socket Docker accorde un accès à l’hôte équivalent à root pour tout ce qui se trouve dans le conteneur ; la page Protect the Docker daemon socket de Docker traite de ce compromis. Sur Docker Desktop pour Windows, le socket de l’hôte est \\.\pipe\docker_engine ; la syntaxe de bind-mount ci-dessus se traduit correctement. Le modèle est général ; il n’a pas été spécifiquement évalué par rapport au sandbox de compétences de zeroclaw dans ce document.

  • Limites de ressources. Docker Desktop sur Windows alloue la RAM/le CPU via %USERPROFILE%\.wslconfig, qui utilise par défaut la moitié de la RAM de l’hôte. L’ensemble des options de configuration est documenté dans la référence Microsoft Advanced settings configuration in WSL. Une enveloppe de départ raisonnable pour un déploiement ZeroClaw mono-utilisateur est :

    [wsl2]
memory=8GB
processors=4

    Augmentez cette valeur si vous exécutez des charges de travail de compétences intensives ou de l’inférence LLM locale dans la même distribution WSL ; il s’agit de recommandations de dimensionnement, et non d’une exigence stricte de l’image.

Dépendances système

Les builds Windows utilisent la chaîne d’outils MSVC. Pour compiler depuis les sources, vous avez besoin de :

  • Outils de build Visual Studio (ou Visual Studio complet) avec la charge de travail « Développement de bureau avec C++ »
  • Rust stable (via rustup)

Si vous utilisez l’Option 1, vous n’avez pas besoin de la chaîne d’outils Rust ; le binaire est autonome. L’option 2 (setup.bat --prebuilt) est censée utiliser le même chemin de binaire, mais le script actuel vérifie toujours la présence de cargo avant d’atteindre la branche prebuilt ; consultez le problème connu ci-dessus.

Exécution en tant que service

Sous Windows, ZeroClaw s’installe en tant que tâche planifiée à portée utilisateur nommée ZeroClaw Daemon. Il n’existe pas d’option Windows Service / LocalSystem dans la version actuelle ; le chemin de code sous-jacent installe toujours une tâche planifiée, que zeroclaw service install soit exécuté depuis un shell avec ou sans élévation de privilèges.

zeroclaw service install
zeroclaw service start

Cela crée une tâche dans le Planificateur de tâches (taskschd.msc) sous votre compte utilisateur qui démarre à l’ouverture de session. Gérez-la via :

zeroclaw service status
zeroclaw service restart
zeroclaw service stop
zeroclaw service logs

À propos de --service-init. La CLI expose un indicateur --service-init [auto|systemd|openrc] pour assurer la cohérence multiplateforme, mais sous Windows il n’a aucun effet ; le chemin de tâche planifiée est toujours utilisé.

Les journaux sont enregistrés dans %USERPROFILE%\.zeroclaw\logs\ (plus précisément, <config_dir>/logs/ où <config_dir> correspond par défaut à %USERPROFILE%\.zeroclaw\). Le wrapper de la tâche planifiée, en revanche, se trouve à côté du fichier de configuration dans %USERPROFILE%\.zeroclaw\zeroclaw-daemon.cmd. Seuls les fichiers de sortie du démon (daemon.stdout.log / daemon.stderr.log) sont écrits dans logs\.

Installations serveur / multi-utilisateurs. La prise en charge native de Windows Service / LocalSystem est prévue dans la feuille de route mais pas encore implémentée. Pour l’instant, sur un serveur, installez ZeroClaw sous le compte sous lequel l’agent doit s’exécuter ; le chemin de tâche planifiée le démarrera à l’ouverture de session de cet utilisateur. Si vous avez besoin qu’il démarre avant l’ouverture de session d’un utilisateur, utilisez Planificateur de tâches → ZeroClaw Daemon → Propriétés → Général → « Exécuter que l’utilisateur soit connecté ou non ».

Mettre à jour

Manuel (chemin de l’option 1)

Réexécutez le bloc d’installation PowerShell de l’Option 1 avec le nouveau $ver. Le nouveau fichier zip remplace le zeroclaw.exe existant sur place. Ensuite :

zeroclaw service restart

setup.bat

Téléchargez à nouveau la dernière version et relancez setup.bat --prebuilt (ou le drapeau que vous avez utilisé initialement). Ensuite :

zeroclaw service restart

Scoop

scoop update zeroclaw       # subject to the staleness caveat above
zeroclaw service restart

Depuis la source

cd C:\path\to\zeroclaw
git pull
cargo install --locked --path . --force
zeroclaw service restart

Désinstaller

Arrêter et supprimer la tâche planifiée :

zeroclaw service stop
zeroclaw service uninstall

Supprimer le binaire :

:Option 1 (préfabriqué manuel) ou setup.bat
rmdir /s /q "%USERPROFILE%\.zeroclaw\bin"

:: Option 3 (cargo install)
del "%USERPROFILE%\.cargo\bin\zeroclaw.exe"

:: Option 4 (Scoop)
scoop uninstall zeroclaw

Supprimer la configuration, l’espace de travail et les journaux (facultatif ; cela supprime l’historique des conversations) :

rmdir /s /q "%USERPROFILE%\.zeroclaw"

La version précédente de ce document faisait référence à %LOCALAPPDATA%\ZeroClaw\ ; ce chemin n’est pas utilisé par la version actuelle ; seul %USERPROFILE%\.zeroclaw\ l’est.

Pièges

  • Chemins longs. Certains systèmes de fichiers Windows limitent encore la longueur des chemins à 260 caractères. Activez la prise en charge des chemins longs si vous rencontrez des erreurs path too long lors d’une compilation depuis les sources :

    reg add HKLM\SYSTEM\CurrentControlSet\Control\FileSystem /v LongPathsEnabled /t REG_DWORD /d 1 /f

  • SmartScreen. Le binaire non signé peut déclencher SmartScreen lors du premier lancement depuis l’Explorateur (double-clic). Clic droit → Propriétés → « Débloquer » est la solution de contournement standard en attendant que nous ajoutions un MSI signé. Le lancement depuis PowerShell ou cmd.exe ne déclenche généralement pas SmartScreen.

  • Arrêt en cas d’inactivité/batterie du Planificateur de tâches. Par défaut, Windows peut interrompre les tâches planifiées en cas d’inactivité ou de fonctionnement sur batterie. La tâche ZeroClaw Daemon installée désactive ces conditions, mais si vous avez effectué l’installation à partir d’une version plus ancienne, vous pouvez le vérifier dans Planificateur de tâches → ZeroClaw Daemon → Propriétés → Conditions :

    • « Démarrer la tâche uniquement si l’ordinateur est sur secteur » : décoché
    • « Arrêter si l’ordinateur passe sur batterie » : décoché
    • « Démarrer la tâche uniquement si l’ordinateur est inactif depuis… » : décoché

  • Authentification par mot de passe OpenSSH. Si vous pilotez Windows via SSH et que l’authentification par clé publique n’est pas acceptée, placez votre clé dans C:\Users\<user>\.ssh\authorized_keys (utilisateur standard) ou C:\ProgramData\ssh\administrators_authorized_keys (lorsque vous êtes connecté en tant que membre du groupe Administrators).
