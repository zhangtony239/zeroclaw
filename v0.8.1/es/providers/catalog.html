ZeroClaw Docs

Catálogo de proveedores

Cada familia de proveedores de modelos que se incluye con ZeroClaw. Para cada una: estructura de configuración, notas sobre autenticación y comportamiento de los endpoints, y la clave de slot que se debe usar en [providers.models.<type>.<alias>].

Consulta Configuration para los campos universales (api_key, uri, model, …) y el orden de resolución.

Los ejemplos a continuación usan home como el alias para subrayar que la mitad del alias la elige el operador; elige el nombre que mejor se ajuste (work, personal, cn, prod, …). Haz referencia a él desde un agente mediante model_provider = "<type>.<alias>".

Nativo

Anthropic: slot anthropic

Admite tokens OAuth (sk-ant-oat*) de suscripciones Claude Pro/Team, sin facturación de API separada. Streaming, llamadas a herramientas, visión y razonamiento están todos soportados. Los endpoints personalizados (proxies compatibles con Anthropic, p. ej. la API Anthropic de Z.AI) también van en este slot: configura uri para sobrescribirlo.

OpenAI: slot openai

Modelos de razonamiento de la serie GPT-4o, GPT-5 y o-series. Los tokens de razonamiento se muestran como eventos ReasoningDelta; consulta Streaming.

OpenAI Codex: slot openai con requires_openai_auth = true

La autenticación de la suscripción de OpenAI Codex reside en el slot openai. Establece wire_api = "responses" para enrutar a través de POST /v1/responses y requires_openai_auth = true para usar el inicio de sesión de la suscripción de Codex (desde el propio ~/.codex/auth.json del Codex CLI) en lugar de un campo api_key en la entrada. La ruta de suscripción no lee OPENAI_API_KEY; esa variable aplica únicamente al modo de clave API openai medido por uso. Consulta Provider Configuration → OAuth and subscription auth para conocer el modelo de credenciales.

Ollama: slot ollama

Inferencia local mediante el /api/chat nativo de Ollama. Salida estructurada basada en esquemas mediante format. Sin clave de API.

Bedrock: ranura bedrock

Gemini: ranura gemini

La API de Gemini de Google. Admite visión y búsqueda fundamentada preejecutada (consulta Streaming para eventos PreExecutedToolCall).

Gemini CLI: slot gemini_cli

Ejecuta el CLI gemini; usa la autenticación existente del CLI.

Azure OpenAI: ranura azure

resource, deployment y api_version residen en esta configuración tipada; no se leen desde variables de entorno.

Copilot: ranura copilot

Utiliza una suscripción de GitHub Copilot para la inferencia del agente. La autenticación usa un token OAuth de Copilot obtenido de GitHub.

Telnyx: slot telnyx

Punto final de IA orientado a la voz. Se combina con el canal clawdtalk para llamadas SIP en tiempo real.

KiloCLI: slot kilocli

Inferencia local a través de KiloCLI.

Kilo AI Gateway: slot kilo

[providers.models.kilo.home]
model   = "anthropic/claude-sonnet-4-6"
api_key = "..."
# endpoint = "gateway"  # predeterminado → https://api.kilo.ai/api/gateway

API en la nube a través de Kilo AI Gateway. Autenticación con token Bearer y múltiples niveles de modelos (free, balanced, pro). El endpoint /models es público (PUBLIC_MODEL_LISTING), por lo que el listado de modelos funciona sin credencial. Como se consulta en vivo, es la fuente que aporta los precios al editor de tarifas de costos. El catálogo compartido de models.dev (clave kilo) es solo una alternativa de respaldo para cuando el endpoint en vivo no está disponible, y no incluye precios.

Migración de nombres: kilo ahora se refiere a este proveedor de gateway. El proveedor de subproceso KiloCLI conserva su espacio kilocli (sinónimo kilo-cli). Si anteriormente configuraste el proveedor de CLI con el atajo kilo, cambia a kilocli.

Todas las ranuras

Cada slot canónico, su endpoint predeterminado, si se ejecuta localmente y su conjunto completo de campos de configuración, generado a partir del registro de proveedores y el esquema de configuración. Haz clic en un slot para expandir sus campos; haz clic en un campo para ver cómo configurarlo. Los slots sin un valor predeterminado fijo necesitan que se establezca uri en la entrada del alias (Azure, custom, familias multirregión, shims de CLI).

Campos compartidos

Cada slot de proveedor acepta estos campos. Los extras específicos de cada slot se enumeran por proveedor a continuación.

api_key 🔑 secret · default —

Token API secreto para este model_provider. Obténgalo desde el panel de control del model_provider (plataforma de OpenAI, consola de Anthropic, página de claves de OpenRouter, etc.). Se almacena mediante el keyring del sistema operativo cuando es posible; nunca lo incluya directamente en config.toml.

Colócalo sobre cualquier superficie:

Panel de control del gateway

Abre /config/providers.models/openrouter y configura el campo providers.models.openrouter.<alias>.api_key.

zerocode

En el panel Config, configure el campo providers.models.openrouter.<alias>.api_key.

zeroclaw config

zeroclaw config set providers.models.openrouter.<alias>.api_key    # entrada enmascarada, almacenada cifrada

fallback string[] · default —

Lista ordenada de otros alias de proveedor que se intentarán cuando todos los modelos de este alias hayan fallado. Cada entrada es una referencia con puntos <type>.<alias> dentro de providers.models y se resuelve con sus propias credenciales, endpoint y modelo. Un fallback nunca hereda la clave de este alias. El recorrido es en profundidad: primero se agotan los modelos de este alias, luego se desciende por cada alias de fallback en orden (aplicando sus propios fallback_models y fallback). Vacío significa que no hay fallback a nivel de proveedor.

fallback_models string[] · default —

Modelos alternativos ordenados para probar en ESTE proveedor antes de recurrir a los alias de fallback. Mismo endpoint, clave y encabezados que el model principal. Solo cambia el identificador del modelo. Usa esto cuando un proveedor ofrece un modelo de respaldo (p. ej., una variante más pequeña o antigua) que debería probarse antes de abandonar el proveedor por completo. Vacío significa que solo se prueba model.

Anula el valor predeterminado del proveedor para las llamadas nativas a herramientas. None (predeterminado) respeta la opción integrada del proveedor. pricing map · default —

Precios por modelo para el seguimiento de costos, en USD por 1M de tokens. Mapa de clave/valor de formato libre. Las claves son identificadores de modelo definidos por el usuario; un sufijo opcional .input / .output codifica la dimensión de precios cuando el operador desea dividir las tarifas. Una clave sin sufijo se usa como tarifa plana por token cuando no se especifica ninguna dimensión. El valor predeterminado es vacío: el seguimiento de costos recurre a tarifas “unknown” y solo se registra el uso de tokens. Ejemplo: pricing = { opus = 15.0, sonnet = 3.0 } O dividido: pricing = { "opus.input" = 15.0, "opus.output" = 75.0 }

Ejemplos prácticos: Morph, GitHub Models, Upstage, Featherless, Arcee, Lambda AI, Inception

Cada uno de estos es un slot estándar compatible con OpenAI: configure model y api_key, omita uri (el endpoint tipado lo proporciona). Ninguno de ellos incluye un índice público de modelos, por lo que el selector de modelos permanece vacío hasta que pegue una credencial. Una vez configurada una clave, ZeroClaw lista los modelos desde el endpoint /models en vivo del proveedor. Los IDs de modelos a continuación son ilustrativos; confirme el catálogo actual en el panel del proveedor.

Morph: slot morph. Modelos rápidos de aplicación de ediciones (morph-v3-large, morph-v3-fast o auto). Clave desde el panel de Morph.

GitHub Models: slot github_models (alias github-models). Modelos de OpenAI / Meta / Microsoft detrás de un único GitHub Personal Access Token. Crea un PAT con el permiso models (de granularidad fina); un token de Copilot no es la misma credencial. Los IDs de modelo llevan el prefijo del editor (p. ej. openai/gpt-4o).

Upstage: slot upstage. Solar Pro / Solar Mini (p. ej. solar-pro2). Clave desde la consola de Upstage.

Featherless: slot featherless. Modelos serverless de pesos abiertos, direccionados por sus IDs de repositorio de Hugging Face (p. ej., meta-llama/Meta-Llama-3.1-8B-Instruct). Clave de featherless.ai.

Arcee: slot arcee. Los modelos nativos incluyen conductor, maestro, virtuoso-large, coder-large y blitz. Clave desde la plataforma Arcee. La API de la plataforma de Arcee usa la ruta base no estándar /api/v1; el endpoint tipado ya tiene esto en cuenta, así que igualmente omita uri.

Lambda AI: slot lambda_ai (alias lambda-ai). Inferencia alojada de Lambda (p. ej. hermes3-405b). Clave desde la página de claves API de Lambda Cloud.

Inception: slot inception. La familia de LLM de difusión Mercury (mercury-coder y el más reciente mercury-2). Clave de la plataforma Inception.

Las credenciales provienen únicamente de la configuración (api_key) o de la anulación --credential en tiempo de ejecución; estas ranuras no leen una variable de entorno *_API_KEY por proveedor.

Ejemplo de NEAR AI Cloud:

[providers.models.nearai.tee]
model   = "..."       # elige un modelId de https://cloud-api.near.ai/v1/model/list
api_key = "..."

El slot nearai usa https://cloud-api.near.ai/v1 de forma predeterminada y envía Authorization: Bearer <api_key>. Para integrar una variable de shell NEARAI_API_KEY existente en la superficie de entorno de réplica de esquema de ZeroClaw, configura ZEROCLAW_providers__models__nearai__tee__api_key="$NEARAI_API_KEY".

Familias multirregión

Varios proveedores chinos exponen distintos endpoints regionales con diferentes modelos predeterminados. Usa una única ranura canónica y selecciona la región con el campo endpoint tipado en la entrada del alias.

Moonshot: ranura moonshot

Variantes: cn, intl, code.

Qwen / DashScope: slot qwen

Las cuentas de Qwen con OAuth usan el mismo slot con auth_mode = "oauth".

GLM: slot glm

MiniMax: slot minimax

Z.AI: slot zai

Para la API compatible con Anthropic de Z.AI, usa [providers.models.anthropic.zai] con uri = "https://api.z.ai/api/anthropic" en su lugar.

Doubao / Volcengine: slot doubao

Los slots restantes de la región de China (yi, hunyuan, qianfan, baichuan) aparecen en la tabla de todos los slots anterior; seleccione la región con el campo endpoint tipado en la entrada del alias.

Capas de enrutamiento

OpenRouter se trata como un único proveedor de primera clase, no como un meta-enrutador. El runtime ve un solo endpoint; OpenRouter gestiona la distribución entre proveedores detrás de ese endpoint.

Para enrutamiento por tarea, ejecuta múltiples agentes y deja que los canales elijan qué agente maneja qué tráfico, consulta Enrutamiento. Para un mecanismo de sugerencias en la configuración más limitado, usa [[model_routes]].

¿Algo falta?

  • Si el endpoint es compatible con OpenAI, usa el slot custom con uri definido.
  • Si tiene su propio slot canónico arriba, usa ese, incluso si solo ves una de sus regiones, el enum endpoint del slot cubre el resto.
  • Si utiliza un formato de protocolo distinto al de OpenAI y necesita su propia implementación, consulte Proveedores personalizados.
